## 一、优化前的问题诊断

### 1.1 原始结构

```
apps/teamclaw/
├── docs/iterators/
│   ├── done/                      # 28 项已完成任务
│   ├── processing/                # 37 项待执行任务
│   ├── poetry-app/                # ⚠️ 25 项 poetry-app 初始构建任务
│   ├── poetry-app-fix/            # ⚠️ 10 项 poetry-app Bug 修复
│   ├── poetry-app-longterm/       # ⚠️ 6 组 poetry-app 长期规划
│   ├── poetry-app-optimize/       # ⚠️ 10 项 poetry-app 优化（与 fix 重复）
│   ├── poetry-app-optimize2/      # ⚠️ 10 项 poetry-app 二轮优化
│   ├── poetry-app-style/          # ⚠️ 5 项 poetry-app 样式任务
│   └── poetry-app-theme/          # ⚠️ 10 项 poetry-app 主题任务
└── issues/
    ├── README.md                  # 索引（定位不清晰）
    ├── 2026-03-24-bugs-xxx.md     # BUG 修复回顾
    └── 2026-03-24-processing-xxx.md # 功能迭代回顾
```

### 1.2 发现的 6 个核心问题

| # | 问题 | 严重程度 | 说明 |
|---|------|----------|------|
| 1 | **跨项目污染** | 🔴 高 | poetry-app 的 7 个子目录（76 个文件）混入 teamclaw 的 `docs/iterators/` |
| 2 | **目录重复** | 🟡 中 | `poetry-app-fix/` 和 `poetry-app-optimize/` 任务名几乎完全相同 |
| 3 | **缺少索引** | 🟡 中 | `processing/`、`done/`、顶层 `iterators/` 均无 README 导航 |
| 4 | **定位混淆** | 🟡 中 | `issues/` 和 `docs/iterators/` 都记录任务，分不清谁是真相源 |
| 5 | **命名不一致** | 🟢 低 | `done/` 用 `【P0】H6` 编号体系，`processing/` 用 `【P0】` 纯优先级 |
| 6 | **缺乏领域分组** | 🟢 低 | 37 项 processing 任务平铺无分类，难以看出功能域和依赖链 |

## 二、优化方案与执行

### 2.1 跨项目隔离

**操作**：将 7 个 poetry-app 目录从 `teamclaw/docs/iterators/` 迁移到 `poetry-app/docs/iterators/`，并简化目录名。

```
# 迁移映射
poetry-app/         →  poetry-app/docs/iterators/v1-initial/
poetry-app-fix/     →  poetry-app/docs/iterators/fix/
poetry-app-optimize/  →  poetry-app/docs/iterators/optimize/
poetry-app-optimize2/ →  poetry-app/docs/iterators/optimize2/
poetry-app-style/   →  poetry-app/docs/iterators/style/
poetry-app-theme/   →  poetry-app/docs/iterators/theme/
poetry-app-longterm/  →  poetry-app/docs/iterators/longterm/
```

**效果**：teamclaw 的 iterators 从 9 个子目录减少到 2 个（`done/` + `processing/`），职责清晰。

### 2.2 添加多层索引

| 位置 | 新增文件 | 作用 |
|------|----------|------|
| `docs/iterators/README.md` | ✅ 新增 | 顶层导航：目录结构、命名规范、生命周期说明 |
| `docs/iterators/processing/README.md` | ✅ 新增 | 37 项任务执行总览：10 个阶段、依赖关系、领域分组 |
| `docs/iterators/done/README.md` | ✅ 新增 | 28 项已完成任务索引：按 H/M/L/Feature 分类 |
| `poetry-app/docs/iterators/README.md` | ✅ 新增 | poetry-app 任务全景：7 个迭代阶段说明 |

### 2.3 明确 issues/ 定位

**改造前**：`issues/README.md` 内容是问题分类表，像另一个任务列表。

**改造后**：明确定义三个目录各自的角色——

```
docs/iterators/processing/  →  前瞻性任务规格书（"要做什么"）
docs/iterators/done/        →  已完成任务归档（"做了什么"）
issues/                     →  迭代回顾总结（"做得怎么样、学到了什么"）
```

## 四、经验教训

### 教训 1：每个 App 独立管理迭代文档

**问题**：在 monorepo 中，把所有 app 的迭代任务都放在一个 app 的 docs 下，导致目录膨胀、职责混乱。

**规则**：
- 每个 `apps/{app-name}/` 维护自己的 `docs/iterators/` 和 `issues/`
- 共享的跨项目文档放在 monorepo 根目录的 `docs/`
- 绝不在 App A 的文档目录下存放 App B 的任务

### 教训 2：目录名要去除冗余前缀

**问题**：`poetry-app-fix/`、`poetry-app-optimize/` 等目录名都带 `poetry-app-` 前缀，在已经位于 teamclaw 文档下时显得冗余且混乱；迁移到 poetry-app 后更无意义。

**规则**：
- 目录名只表达**功能语义**，不重复父级路径信息
- 好：`fix/`、`optimize/`、`theme/`
- 坏：`poetry-app-fix/`、`poetry-app-optimize/`

### 教训 3：尽早建立索引，不要等文件多了再补

**问题**：37 个任务文件平铺在 `processing/` 中，没有 README 索引，每次要找任务都得逐个翻阅文件名。

**规则**：
- 目录超过 **5 个文件**时，必须有 README.md 索引
- 索引应包含：全景表格、分类/分组、依赖关系
- 索引应随任务状态变化同步更新

### 教训 4：分清"任务管理"与"迭代回顾"

**问题**：`issues/` 目录的内容实际上是迭代结束后的回顾报告，但它的 README 写成了另一个任务列表的样子，与 `docs/iterators/processing/` 的职责重叠。

**规则**：
```
任务规格书 (iterators/processing/)
  - 面向未来："接下来要做什么"
  - 包含：需求描述、技术方案、依赖关系、实现文件列表
  - 生命周期：创建 → 进行中 → 完成归档

迭代回顾 (issues/)
  - 面向过去："这轮做得怎么样"
  - 包含：完成了什么、跳过了什么、遇到的问题、经验教训
  - 每轮迭代结束后生成一份
```

### 教训 5：避免重复迭代目录

**问题**：`poetry-app-fix/` 和 `poetry-app-optimize/` 的 10 个任务标题几乎完全相同（"检查并修复页面加载性能问题"、"修复搜索功能潜在问题"等），只是文件大小略有差异。这是两次独立对话中生成的重复内容。

**规则**：
- 新建迭代目录前，先检查是否已有同主题的目录
- 如果需要追加/更新，在原目录内修改，而非创建新目录
- 命名加入**版本号或日期**以区分：`optimize-v1/`、`optimize-v2/`（而非 `optimize/`、`optimize2/`）

### 教训 6：统一命名规范

**问题**：
- `done/` 文件用 `【P0】H6 Agent实际LLM执行接入` 的 `H/M/L/F + 序号` 体系
- `processing/` 文件用 `【P0】API Token 数据模型与 CRUD 接口` 的纯描述体系
- 两种体系并存，增加认知负担

**规则**：全项目统一一种命名格式——

```
{两位序号}【{优先级}】{简洁描述}.md
```

- 序号：执行优先级（01 = 最优先）
- 优先级：P0 / P1 / P2 / P3
- 描述：中文，简洁表意，20 字以内

### 教训 7：AI 对话生成的文档需要人工审阅归档

**问题**：多次 AI 对话各自生成了独立的任务目录（`poetry-app-fix`、`poetry-app-optimize`），没有回顾已有结构，导致重复和散乱。

**规则**：
- AI 生成的任务文档视为**草稿**，需要人工审阅后归档到正确位置
- 每次让 AI 生成迭代计划时，**先把已有目录结构作为上下文传入**
- 定期（每周/每个迭代结束时）做一次目录治理

## 五、推荐的目录管理流程

```
1. 需求/Bug 进入
   └→ 在 processing/ 中创建任务文件
   └→ 更新 processing/README.md 索引

2. 任务执行中
   └→ 在任务文件中记录进展
   └→ 更新状态标记（⏳ → 🔧）

3. 任务完成
   └→ 移动到 done/
   └→ 更新 done/README.md 索引
   └→ 更新 processing/README.md（标记 ✅）

4. 迭代结束
   └→ 在 issues/ 生成回顾总结
   └→ 包含：完成项、跳过项、经验教训、数据统计

5. 定期治理（每 1-2 周）
   └→ 检查目录是否有跨项目文件
   └→ 检查重复/过时的目录
   └→ 确保 README 索引与实际文件一致
```
