# 迭代任务与回顾管理规范

> 适用范围：致富经 Monorepo 下所有 App（teamclaw、poetry-app 及后续新增项目）
> 版本：v1.0
> 生效日期：2026-03-26

本文档定义 Agent 自动迭代过程中**任务记录、生命周期流转、迭代回顾**的目录结构与操作规范。所有 Agent（包括 Cascade、Coder、PM 等）在执行迭代时必须遵守本规范。

---

## 一、核心概念

本规范管理两类文档，分别承载不同职责：

| 文档类型 | 目录 | 面向 | 核心问题 |
|----------|------|------|----------|
| **任务规格书** | `docs/iterators/` | 未来 | "接下来要做什么、怎么做" |
| **迭代回顾** | `issues/` | 过去 | "这轮做得怎么样、学到了什么" |

两者**不可混用**：`iterators/` 不写回顾，`issues/` 不写任务规格。

---

## 二、目录结构标准

### 2.1 每个 App 的标准目录

每个 `apps/{app-name}/` 下**必须**独立维护以下两个目录：

```
apps/{app-name}/
├── docs/
│   └── iterators/               # 任务生命周期管理
│       ├── README.md            # 索引（必须）
│       ├── processing/          # 待执行 & 进行中
│       │   └── README.md        # 任务总览（必须）
│       └── done/                # 已完成归档
│           └── README.md        # 完成索引（必须）
└── issues/                      # 迭代回顾报告
    └── README.md                # 索引（必须）
```

### 2.2 隔离原则

- **App 之间任务严格隔离**：App A 的任务只能出现在 `apps/A/docs/iterators/` 下，禁止存放到 App B 的目录中
- **跨 App 的共享文档**：放在 monorepo 根目录 `docs/`，不属于任何单一 App
- 如需引用其他 App 的任务，使用相对路径链接，不复制文件

### 2.3 README 索引要求

任何包含 **3 个以上文件**的目录，必须有 `README.md` 索引。索引应包含：

| 必选内容 | 说明 |
|----------|------|
| 目录用途说明 | 一句话定位 |
| 文件列表表格 | 编号、优先级、名称、状态、依赖 |
| 领域分组 | 按功能域对任务分类 |

## 三、任务规格书规范（`docs/iterators/`）

### 3.1 任务文件命名

```
{两位序号}【{优先级}】{简洁描述}.md
```

| 字段 | 规则 | 示例 |
|------|------|------|
| 序号 | `01`-`99`，表示执行优先级，`01` 最优先 | `05` |
| 优先级 | `P0` 阻塞 / `P1` 重要 / `P2` 改进 / `P3` 锦上添花 | `P0` |
| 描述 | 中文优先，简洁表意，≤ 20 字 | `API Token 数据模型与 CRUD 接口` |

完整示例：`05【P0】API Token 数据模型与 CRUD 接口.md`

### 3.2 任务文件内容模板

```markdown
# {序号}【{优先级}】{任务名称}

## 背景
{为什么需要做这件事，1-3 句话}

## 目标
{做完后达到的效果，用列表列出可验证的目标}

## 技术方案
{实现思路、数据结构、API 设计等}

## 实现文件
- `path/to/file1.ts` — 说明
- `path/to/file2.ts` — 说明

## 依赖关系
- 前置：{依赖哪些任务完成}
- 后续：{哪些任务依赖本任务}

## 验证方式
{如何确认任务完成：命令、URL、预期输出}

## 状态
⏳ 待执行 / 🔧 进行中 / ✅ 已完成
```

### 3.3 任务生命周期

```
创建任务
  │  在 processing/ 中新建文件，状态标记 ⏳
  │  更新 processing/README.md 索引
  ▼
开始执行
  │  状态改为 🔧
  │  在文件中追加进展记录
  ▼
完成任务
  │  状态改为 ✅
  │  在文件末尾追加完成信息（时间、Commit、说明）
  │  将文件移动到 done/
  │  更新 done/README.md 和 processing/README.md
  ▼
归档
```

### 3.4 processing/README.md 标准结构

```markdown
# {App 名称} — 待执行任务总览

> 共 N 项任务，按执行优先级排序

## 执行顺序 & 依赖关系

### 阶段一：{阶段名称}（{序号范围}）

| # | 优先级 | 任务 | 依赖 | 状态 |
|---|--------|------|------|------|
| 01 | P0 | xxx | — | ✅ |
| 02 | P0 | xxx | → 01 | ⏳ |

### 阶段二：...

## 领域分组

| 领域 | 任务编号 |
|------|----------|
| Token 管理 | 05, 07, 09 |
| 前端样式 | 23, 24, 25 |
```

## 四、迭代回顾规范（`issues/`）

### 4.1 定位

`issues/` 目录存放**每轮迭代结束后的回顾报告**。它记录的是"发生了什么"，而非"要做什么"。

回顾报告的价值：
- 记录实际执行中遇到的障碍和绕行方案
- 沉淀可复用的技术决策和调试经验
- 为后续迭代提供参考基线

### 4.2 回顾文件命名

```
{YYYY-MM-DD}-{迭代主题}-issues.md
```

| 字段 | 规则 | 示例 |
|------|------|------|
| 日期 | ISO 格式，迭代结束日期 | `2026-03-25` |
| 主题 | 英文短名，用 `-` 连接 | `v1-initial`、`bugs`、`token-management` |

完整示例：`2026-03-25-token-management-issues.md`

### 4.3 回顾文件内容模板

```markdown
# {App 名称} {迭代主题} 迭代回顾

> 日期：{YYYY-MM-DD}
> 迭代轮次：N 轮
> 任务范围：processing/ 中的 #{起始}-#{结束}

## 完成概况

| 指标 | 数值 |
|------|------|
| 计划任务数 | X |
| 实际完成数 | Y |
| 跳过/延期数 | Z |
| 新增文件数 | N |
| 修改文件数 | M |

## 逐项记录

### {序号}【{优先级}】{任务名称}
- **状态**：完成 / 跳过 / 部分完成
- **Commit**：{hash}（如无则填 —）
- **执行摘要**：{1-2 句话描述做了什么}
- **遇到的问题**：{如有}
- **解决方案**：{如有}

## 技术决策记录

{本轮迭代中做出的重要技术决策及理由}

## 待改进项

{下轮迭代需要注意的事项}
```

### 4.4 issues/README.md 标准结构

```markdown
# {App 名称} 迭代回顾索引

> 本目录存放每轮迭代结束后的回顾总结

## 目录定位

- `docs/iterators/processing/` → 任务规格书（"要做什么"）
- `docs/iterators/done/` → 完成归档（"做了什么"）
- `issues/` → 迭代回顾（"做得怎么样"）

## 回顾文件列表

| 日期 | 文件 | 迭代主题 | 任务范围 |
|------|------|----------|----------|
| 2026-03-24 | xxx-issues.md | Bug 修复 | #01-#28 |
| 2026-03-25 | xxx-issues.md | Token 管理 | #05-#12 |
```

## 五、Agent 自动化操作规范

以下规则约束所有 Agent 在自动迭代中的文档操作行为。

### 5.1 创建任务前

1. **先读取目标 App 的 `docs/iterators/processing/README.md`**，了解已有任务和编号
2. **检查是否已有同主题任务**，如有则更新而非新建
3. **确认任务属于哪个 App**，放入对应 App 的 `docs/iterators/`

### 5.2 创建任务时

1. 在 `processing/` 中创建文件，遵循命名规范 `{序号}【{优先级}】{描述}.md`
2. 使用内容模板填写，所有字段不可留空（无依赖写 `—`）
3. 同步更新 `processing/README.md` 的任务表格

### 5.3 完成任务时

1. 在任务文件末尾追加完成信息：

```markdown
## ✅ 已完成

**完成时间**：{YYYY-MM-DD HH:mm} ({时区})
**Commit**：{hash}
**说明**：{一句话总结}
```

2. 将文件从 `processing/` 移动到 `done/`
3. 更新 `processing/README.md`（状态改为 ✅）和 `done/README.md`（新增条目）

### 5.4 迭代结束时

1. 在 `issues/` 中生成回顾报告，遵循回顾文件命名和内容模板
2. 更新 `issues/README.md` 的文件列表

### 5.5 禁止行为

| 禁止 | 原因 |
|------|------|
| 在 App A 的目录下创建 App B 的任务 | 跨项目污染 |
| 创建与已有任务高度重复的文件 | 信息冗余 |
| 创建目录时使用父级名称作为前缀 | 命名冗余（如 `poetry-app-fix/`） |
| 在 `issues/` 中写任务规格书 | 职责混用 |
| 在 `iterators/` 中写迭代回顾 | 职责混用 |
| 创建超过 3 个文件的目录而不建 README | 缺少索引 |

## 六、迭代批次目录规范（可选）

当一个 App 存在多轮主题性迭代时（如 poetry-app 的构建→修复→优化→主题），可在 `iterators/` 下按批次建子目录：

```
docs/iterators/
├── README.md
├── processing/           # 当前活跃任务（始终存在）
├── done/                 # 完成归档（始终存在）
├── v1-initial/           # 可选：V1 构建批次归档
├── fix/                  # 可选：Bug 修复批次归档
└── optimize-v1/          # 可选：优化第一轮归档
```

批次目录命名规则：
- 使用**功能语义名**，不带 App 名前缀
- 多轮同主题使用 `-v1`、`-v2` 后缀区分
- 每个批次目录内也需要 `README.md` 索引

---

## 七、目录健康检查清单

每轮迭代结束或每周执行一次：

- [ ] 每个 App 的 `iterators/` 和 `issues/` 目录是否独立存在
- [ ] `processing/README.md` 中的状态是否与文件实际状态一致
- [ ] `done/README.md` 是否包含所有已完成文件
- [ ] `issues/README.md` 的文件列表是否完整
- [ ] 是否存在跨 App 的任务文件
- [ ] 是否存在高度重复的任务/目录
- [ ] 所有超过 3 个文件的目录是否有 README
