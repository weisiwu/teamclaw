# 项目导入模块

[← 返回主文档](../系统架构.V1.md)

---

## 1. 目标

将业务代码导入系统，通过多种机制解析项目，预先生成记忆、RAG、Skills，加速后续任务执行。

---

## 2. 数据源要求

> ⚠️ **严格禁止使用 mock 数据或伪造数据。** 所有模块的数据必须来自真实项目仓库。
>
> 🎨 **样式检测要求**：每个功能在实现过程中，必须检测并验证实际渲染的 UI 样式（布局、间距、配色、响应式等），确保页面展示效果与设计一致，禁止只关注逻辑而忽略样式。

当前默认数据源为 GitHub 仓库：**https://github.com/weisiwu/teamclaw**

用户可在项目导入时选择两种数据获取方式：

| 方式 | 说明 | 适用场景 |
|------|------|---------|
| **URL 下载** | 填写 Git 仓库 URL，系统自动 clone/pull | 远程协作、CI/CD 环境、仓库在 GitHub/GitLab 等平台 |
| **本地上传** | 指定本地磁盘路径，直接读取 | 内网开发、无外网环境、本地有最新代码 |

两种方式获取的项目内容完全等价，后续解析流程一致。切换数据源不影响已生成的记忆和 Skills（重新导入时会增量更新）。

---

## 3. 导入入口

用户通过 **Web 后台管理平台** 发起项目导入：
1. 启动内网 Web 服务
2. 选择数据源方式（URL 下载 或 本地路径）
3. 填写仓库 URL 或指定本地项目路径
4. 系统开始初始化解析
5. 后续通过定时任务刷新 或 手动强制刷新

---

## 4. 导入流程（11 步）

```
项目路径输入
    │
    ├── 1. 扫描文件结构，生成目录树
    ├── 2. 识别技术栈（框架、语言、构建工具）
    ├── 3. 解析文档（PPT/Word/Excel/Text/Markdown）
    ├── 4. 梳理代码架构（模块划分、依赖关系）
    ├── 5. 了解打包/编译机制
    ├── 6. 生成项目摘要（≤500 字）
    ├── 7. 生成功能定位文件（页面/功能拆分）
    ├── 8. 自动生成 Skills（供所有 Agent 使用）
    ├── 9. 将文档转换为在线文档，供浏览
    ├── 10. 向量化存储到记忆数据库（RAG）
    └── 11. 基于历史改动，建立记忆共享、反馈增强机制
```

---

## 5. 核心机制：上下文压缩与快速检索

### 5.1 问题

如何将很大的上下文压缩成摘要，并通过摘要快速对应原文？

### 5.2 解决方案：四层检索架构

| 层次 | 存储内容 | 作用 |
|------|---------|------|
| **摘要层** | ≤500 字项目摘要 | 快速理解项目整体 |
| **索引层** | 摘要 → 原文映射（文件路径 + 行号） | 通过摘要定位原文 |
| **向量层** | 代码/文档的 Embedding | RAG 检索相关片段 |
| **原文层** | 完整代码和文档 | 详细查看 |

### 5.3 检索流程

```
用户任务 → 提取关键词 → 向量检索（召回 Top 5-10）
         → 读取摘要 → 定位原文 → 补充上下文
```

---

## 6. 记忆策略

| 策略 | 说明 |
|------|------|
| **项目摘要常驻** | 导入时生成 ≤500 字摘要，作为所有任务的基础上下文 |
| **增量记忆** | 每次任务完成只追加变更摘要，不重复存完整对话 |
| **RAG 按需召回** | 根据任务关键词，从向量数据库召回最相关的 5-10 条记忆 |
| **摘要→原文映射** | 每条摘要保留原文文件路径和行号，需要时可回溯 |
| **定时刷新** | 内容更新后通过定时任务或手动触发重新解析 |

---

## 7. 自动 Skill 生成

项目导入后，系统分析代码自动生成可复用的 Skills，存放到共享目录：

```
~/.openclaw/skills/
  ├── project-{name}-build.md      # 构建指南
  ├── project-{name}-structure.md   # 代码结构
  ├── project-{name}-deploy.md      # 部署流程
  └── project-{name}-test.md        # 测试规范
```

---

## 8. 模型分级

| 级别 | 用途 | 模型 |
|------|------|------|
| **轻量** | 文件扫描、文档解析、摘要生成 | DeepSeek-Chat / Kimi |
| **中等** | 代码架构分析、RAG 向量化 | GPT-4o-mini / DeepSeek-V3 |
| **强力** | 复杂代码理解、Skill 生成 | Claude Sonnet / GPT-4o |

---

## 9. 文档转换

项目中的文档（PPT/Word/Excel/Markdown）会被转换为在线可浏览格式：

- **PPT** → HTML 幻灯片
- **Word/Excel** → HTML 表格/文档
- **Markdown** → 渲染后的 HTML
- **代码文件** → 语法高亮的 HTML

所有转换后的文档存储在：`~/.openclaw/docs/{project_name}/`

---

## 10. 历史改动与反馈增强

基于 Git 历史改动，建立记忆共享和反馈增强机制：

1. **分析 Git Log**：提取每次 commit 的改动摘要
2. **关联任务**：将 commit 与任务关联
3. **建立模式**：识别常见的改动模式（如"修改样式"、"修复 Bug"）
4. **反馈增强**：后续类似任务时，自动召回相关历史改动作为参考

## 11. API 定义

### 11.1 项目管理 API

| 方法 | 路径 | 说明 |
|------|------|------|
| `POST` | `/api/v1/projects/import` | 发起项目导入（传入 URL 或本地路径） |
| `GET` | `/api/v1/projects/import/:taskId/status` | 查询导入进度（11 步中的第几步） |
| `GET` | `/api/v1/projects` | 获取已导入的项目列表 |
| `GET` | `/api/v1/projects/:projectId` | 获取项目详情（摘要、技术栈、统计） |
| `POST` | `/api/v1/projects/:projectId/refresh` | 手动触发项目重新解析 |
| `DELETE` | `/api/v1/projects/:projectId` | 删除已导入项目 |

### 11.2 关键请求/响应示例

**POST /api/v1/projects/import**

```json
// 请求
{
  "source": "url",            // "url" 或 "local"
  "url": "https://github.com/weisiwu/teamclaw",
  "localPath": null,           // source 为 "local" 时填写
  "name": "teamclaw"           // 可选，默认从仓库名提取
}

// 响应
{
  "code": 0,
  "data": {
    "taskId": "import_20260317_001",
    "projectId": "proj_teamclaw",
    "status": "processing",
    "currentStep": 1,
    "totalSteps": 11
  },
  "message": "ok"
}
```
