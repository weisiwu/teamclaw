## 🔴 高优先级（架构/数据安全）

### H1. 数据存储混乱：SQLite + PostgreSQL + 内存三套并存

**现状**：

| 存储方式 | 使用位置 | 存储内容 |
|---------|---------|---------|
| SQLite（better-sqlite3） | `server/src/db/sqlite.ts` | 版本、标签、分支、审计日志、搜索历史、下载任务 |
| PostgreSQL（pg） | `server/src/utils/db.ts` | 连接池已创建，但几乎没有路由实际使用 |
| 内存 Map | `taskLifecycle.ts`、`messageQueue.ts`、`importOrchestrator.ts` | 任务、消息队列、导入任务 |

**风险**：
- 服务重启后，所有任务和消息队列数据**全部丢失**
- SQLite 和 PostgreSQL 定位重复，维护成本翻倍
- docker-compose 中配置了 PostgreSQL 但实际未使用

**优化方案**：
1. 将 SQLite 中的表全部迁移到 PostgreSQL
2. 任务和消息队列改为 PostgreSQL 持久化 + Redis 缓存
3. 导入任务状态持久化到 PostgreSQL
4. 移除 better-sqlite3 依赖

**涉及文件**：
- `server/src/db/sqlite.ts` → 废弃
- `server/src/db/migrations/run.ts` → 重写为 PostgreSQL 迁移
- `server/src/services/taskLifecycle.ts` → 增加 DB 持久化
- `server/src/services/messageQueue.ts` → 增加 DB 持久化
- `server/src/services/importOrchestrator.ts` → 增加 DB 持久化
- `server/src/routes/version.ts` → 改用 PostgreSQL 查询

### H2. 认证机制极其薄弱——任何人可伪造管理员

**现状**：

`server/src/middleware/auth.ts` 中的认证完全依赖客户端传入的 HTTP header：

```
X-User-Id: user_001
X-User-Role: admin
```

没有任何签名、加密或验证机制，**任何人伪造 header 即可获得管理员权限**。

**风险**：
- 所有管理员 API（删除项目、修改配置、定时任务）可被任意调用
- docker-compose 中已配置 `JWT_SECRET` 但代码中从未使用

**优化方案**：
1. 实现 JWT 认证流程：登录 → 签发 Token → 请求携带 Bearer Token
2. auth 中间件从 Token 中解析用户身份，而非信任 header
3. 在 `server/src/routes/user.ts` 中增加登录端点 `POST /api/v1/auth/login`
4. 前端 `lib/api/` 中统一在请求头带上 `Authorization: Bearer <token>`

**涉及文件**：
- `server/src/middleware/auth.ts` → 重写为 JWT 验证
- `server/src/routes/user.ts` → 增加登录/注销端点
- `server/src/services/userService.ts` → 增加密码哈希、Token 签发
- `lib/api-shared.ts` → 前端请求统一携带 Token
- `middleware.ts`（Next.js）→ 可增加 Token 校验

### H3. version.ts 路由文件 80KB / 2305 行，严重违反单一职责

**现状**：

`server/src/routes/version.ts`（80KB）是全项目最大的单文件，包含：
- 版本 CRUD
- Git Tag 管理
- 构建触发与产物管理
- 版本回退
- 版本对比
- 截图管理
- 版本摘要
- Bump 历史
- 版本设置
- 类型定义
- 迁移执行（`runMigrations()` 在顶部直接调用）

**风险**：
- 代码不可维护，改一处容易引发其他功能回归
- 路由文件中混入了类型定义和业务逻辑
- 迁移在路由模块加载时执行，不可控

**优化方案**：

拆分为以下文件：

| 新文件 | 内容 | 预估行数 |
|-------|------|---------|
| `routes/version.ts` | 版本 CRUD（创建、列表、详情、更新、删除） | ~200 |
| `routes/versionBuild.ts` | 构建触发、重新构建、构建配置 | ~200 |
| `routes/versionRollback.ts` | 回退、回退预览、回退历史 | ~200 |
| `routes/versionTag.ts` | Tag 创建、列表、删除、自动 Tag | ~200 |
| `routes/versionCompare.ts` | 版本对比 | ~100 |
| `routes/versionScreenshot.ts` | 截图上传、关联、删除 | ~150 |
| `routes/versionSummary.ts` | 摘要生成、编辑 | ~100 |
| `routes/versionSettings.ts` | 版本设置、Bump 配置 | ~100 |
| `models/version.ts`（已存在但不完整） | 所有版本相关类型定义 | ~100 |

同理，前端 `lib/api/versions.ts`（96KB / 3231 行）也需按相同维度拆分。

### H5. 导入流程异步编排串联——11 步流水线断链（**新增**）

**现状**：

项目导入模块文档定义了 11 步导入流水线，各步骤的服务文件大多已实现，但 **没有被串联到导入路由中**：

| 层级 | 问题 |
|------|------|
| `server/src/routes/project.ts` POST /import | 仅同步执行 clone → scan → detectTechStack（内联简化版）3 步后立即返回，剩余 8 步从未执行 |
| `server/src/services/importOrchestrator.ts` | 已编写完整状态管理逻辑（createImportTask/executeStep/completeStep），但**从未被路由导入** |
| `app/import/page.tsx` 前端进度页 | 轮询后端进度，但后端 step 4 永远 `pending`，用户看到无限等待 |

已实现但未被调用的服务：`docParser.ts`、`codeAnalyzer.ts`、`summaryGenerator.ts`、`skillGenerator.ts`、`featureMap.ts`、`docConverter.ts`、`vectorStore.ts`、`gitHistoryAnalysis.ts`。

**风险**：
- 导入流程无法真正完成，前端永远卡在进度页
- 8 个已编写的服务文件成为死代码
- 功能定位、Skills 生成、向量化等核心能力全部不可用

**优化方案**：
1. 重写 `importOrchestrator.ts` 为流程执行器，每步绑定真实服务调用
2. `POST /import` 改为：创建任务 → 立即返回 → 后台异步执行全部步骤
3. 每步完成后更新 task.steps 状态，前端轮询自动推进
4. 非关键步骤（vectorize、generateSkills）失败时标记 warning 但继续

**涉及文件**：
- `server/src/services/importOrchestrator.ts` → 重写为流程执行器
- `server/src/routes/project.ts` → POST /import 改为异步编排；移除内联 helper
- `app/import/page.tsx` → STEP_LABELS 与后端对齐
- 新建 `server/src/services/buildDetector.ts`（打包机制检测，见 M9）

### H6. Agent 实际 LLM 执行接入——协作机制空壳（**新增**）

**现状**：

Agent 管理框架完整（运行时管理、派发规则、健康监控、执行引擎），但 `agentExecution.ts` 的 `dispatchToAgent()` 创建执行记录后**立即返回**，从未调用 `llmService.ts` 或任何外部 LLM/OpenClaw 接口：

| 组件 | 状态 |
|------|------|
| `agentService.ts` — 运行时管理 | ✅ 配置查询、状态追踪、负载均衡 |
| `agentExecution.ts` — 执行引擎 | ⚠️ 只创建 ExecutionContext，**不调用 LLM** |
| `dispatchService.ts` — 派发规则 | ✅ 等级校验、权限检查 |
| `llmService.ts` — LLM 调用 | ✅ 三级模型 + 自动路由（从未被 Agent 调用） |

**风险**：
- Agent 永远不会真正执行任务，coder 无法写代码、reviewer 无法审代码
- `llmService.ts` 的三级模型和自动路由成为死代码
- 整个 Agent 协作只是状态管理的空壳

**优化方案**：
1. 重写 `dispatchToAgent` 为异步执行 + 真实 LLM 调用
2. 为 5 个 Agent 角色创建 system prompt 模板
3. LLM 调用前注入项目记忆 + 任务上下文 + 向量检索结果
4. main Agent 响应后自动派发子任务给 coder/reviewer

**涉及文件**：
- `server/src/services/agentExecution.ts` → 核心改造
- 新建 `server/src/prompts/agentPrompts.ts` → 角色 prompt 模板

### H7. 消息→任务→Agent 流水线串联——三大模块断裂（**新增**）

**现状**：

消息、任务、Agent 三大模块各自实现丰富内部逻辑，但**模块之间完全断裂**：

| 断裂点 | 现状 |
|--------|------|
| 消息 → 任务 | 消息入队后只改队列状态，从不创建任务 |
| 消息 → Agent | `channelAdapter` 检测到 @agent 后丢弃信息，不触发执行 |
| 任务 → Agent | 任务状态变更不触发 Agent 派发 |
| Agent → 消息回复 | 执行完成后不回复消息通道 |
| 任务完成 → 版本 | `autoBump` 存在但未注册为 taskLifecycle 钩子 |

**风险**：
- 消息石沉大海，用户收不到任何响应
- 必须手动通过 API 创建任务和触发 Agent
- 系统无法按架构文档的「消息驱动」模式运行

**优化方案**：
1. 创建 `messagePipeline.ts` 消息处理主管道（@检测 → 权限 → 任务创建 → Agent 派发）
2. POST /messages 接入管道
3. Agent 执行完成后回复消息通道
4. 注册 taskLifecycle 钩子 → autoBump + 群聊通知
5. 消息入队后立即发送「已收到」回复
6. 抢占触发时暂停当前 Agent 执行

**涉及文件**：
- 新建 `server/src/services/messagePipeline.ts`
- 新建 `server/src/services/messageReply.ts`
- 新建 `server/src/services/taskLifecycleHooks.ts`
- `server/src/routes/message.ts` → 接入管道
- `server/src/services/agentExecution.ts` → 完成回调
- `server/src/services/preemptionService.ts` → 联动 Agent 暂停

## 🟠 中优先级（可靠性/可维护性）

### M1. 数据库迁移不规范——无版本追踪

**现状**：

`server/src/db/migrations/run.ts` 中用以下模式做列添加：

```typescript
try {
  db.prepare("ALTER TABLE versions ADD COLUMN git_tag TEXT").run();
} catch (e: unknown) {
  // Column may already exist, ignore
}
```

重复约 10 次。没有迁移版本表，无法知道哪些迁移已执行。

**优化方案**：
1. 创建 `migrations` 表，记录已执行的迁移 ID 和时间
2. 每个迁移文件有唯一 ID（如时间戳），执行前检查是否已记录
3. 迁移执行幂等，避免重复

---

### M2. Next.js API Routes 与 Express Server 功能重复

**现状**：

| 层 | 路径前缀 | 文件数量 | 说明 |
|----|---------|---------|------|
| Next.js API Routes | `app/api/v1/` | 32 个子目录 | 有些直接处理业务（如 dashboard/overview），有些 proxy 到 Express |
| Express Server | `server/src/routes/` | 23 个文件 | 完整的后端 API |

前端代码有时调用 Next.js route（如 `/api/v1/dashboard/overview`），有时直接调 Express（如通过环境变量 `SERVER_HOST`）。

**风险**：
- 同一个 API 可能有两套实现逻辑
- 数据源可能不一致（Next.js 中用内存 store，Express 中用 SQLite/DB）

**优化方案**：
- 明确 Next.js API Routes **仅**作为 proxy 层，统一转发到 Express
- 或移除 Next.js API Routes，前端直接调 Express 后端（开发时通过 `next.config.js` 的 `rewrites` 代理）

### M3. Git Clone 存在命令注入风险

**现状**：

```typescript
// server/src/services/gitClone.ts:23
await execAsync(`git clone ${url} "${destPath}" --depth 1`);
```

`url` 参数未做任何过滤就直接拼入 shell 命令。攻击者可传入 `; rm -rf /` 等恶意内容。

**优化方案**：
1. 使用 `execFile('git', ['clone', url, destPath, '--depth', '1'])` 替代 `exec`
2. 增加 URL 格式校验：必须匹配 `https://` 或 `git@` 开头
3. 禁止包含 `;`、`|`、`&`、`$` 等 shell 元字符

---

### M4. 向量化服务缺少批量处理和错误恢复

**现状**：

`server/src/services/vectorStore.ts` 中逐条调用 Embedding API：

```typescript
for (const text of texts) {
  const emb = await getEmbedding(text, apiKey, baseUrl);
  embeddings.push(emb);
}
```

**风险**：
- 大量文档时效率极低（逐条 HTTP 请求）
- 中间某条失败则前面的结果浪费
- 无重试机制

**优化方案**：
1. 改为批量 Embedding（OpenAI API 支持数组 input，单次最多 2048 条）
2. 增加指数退避重试（429 限流时自动重试）
3. 增加分批处理，每批 100 条，失败的批次单独重试

---

### M5. Redis / PostgreSQL 连接无优雅关闭

**现状**：

`server/src/utils/redis.ts` 和 `server/src/utils/db.ts` 创建了连接，但进程退出时无 graceful shutdown。

**优化方案**：

在 `server/src/index.ts` 中增加：

```typescript
process.on('SIGTERM', async () => {
  console.log('Graceful shutdown...');
  await pool.end();       // PostgreSQL
  redis.disconnect();     // Redis
  closeDb();              // SQLite (临时)
  process.exit(0);
});
```

### M6. 错误响应格式不统一

**现状**：

项目中存在两种错误响应格式：

| 来源 | 格式 |
|------|------|
| `middleware/errorHandler.ts` | `{ error: true, code: "ERROR_CODE", message, requestId, timestamp }` |
| 路由中直接 `res.json()` | `{ code: 400, data: null, message }` |

前端需要同时处理两种结构。

**优化方案**：
1. 统一为一种格式（建议用 errorHandler 的格式，更完整）
2. 所有路由中的错误响应都通过 `throw new ApiError(...)` 抛出，由全局 handler 统一处理
3. 前端 `lib/api-shared.ts` 中统一错误解析逻辑

### M7. 项目中存在大量 Mock / 伪造数据

**现状**：

项目文档明确要求「严格禁止使用 mock 数据或伪造数据」，但代码中存在大量违规：

| 层级 | 文件数 | 说明 |
|------|--------|------|
| 前端 API 层（`lib/api/`） | 4 个文件，12 个 mock 数组 | `versions.ts`、`tasks.ts`、`branches.ts`、`tags.ts` 全量 mock，API 失败时静默回退假数据 |
| 后端服务层（`server/src/services/`） | 46 个文件用内存 Map | `tokenStatsService.ts` 启动时生成 30 天假数据，`userService.ts` 硬编码 3 个用户 |
| 前端组件层（`components/`） | 4 个组件 | `MessageSelector.tsx` 含 25 条假消息，`BranchCompareDialog.tsx` 全量伪造对比数据 |
| 前端路由 API（`app/api/v1/`） | 3 个路由 | `build/stats`、`build/trigger` 硬编码返回，`settings` 角色硬编码 admin |

**风险**：
- 前端页面展示伪造内容，无法验证真实功能
- API 调用失败时静默回退 mock，掩盖真实 bug
- Token 统计等监控数据不真实，误导决策

**优化方案**：
1. 删除前端 `lib/api/` 中所有 `mock*` 数组和 `delay()` 模拟，API 失败时抛出错误而非回退 mock
2. 删除后端 `initFakeData()`、`initSeedData()` 等假数据生成函数
3. 种子数据移至 `scripts/db-seed.sh`，仅开发环境使用
4. 未配置外部服务时明确提示「未配置」而非展示假数据
5. 前端所有页面补全加载态、空状态、错误态

**涉及文件**：
- `lib/api/versions.ts`、`tasks.ts`、`branches.ts`、`tags.ts` → 删除 mock 数组
- `components/versions/MessageSelector.tsx` → 删除 mockMessages
- `components/branch/BranchCompareDialog.tsx` → 删除 mockCompareData
- `server/src/services/tokenStatsService.ts` → 删除 initFakeData
- `server/src/services/userService.ts` → 删除 initSeedData
- `server/src/routes/feishu.ts` → 未配置时返回 503
- `app/api/v1/build/stats/route.ts`、`trigger/route.ts` → 代理到真实后端

### M10. 消息通道双向对接——飞书发送 + 微信接入（**新增**）

**现状**：

架构文档要求接入微信和飞书消息通道。当前只实现了「接收」适配，缺少「发送」能力和微信实际接入：

| 通道 | 接收（入站） | 发送（出站） |
|------|------------|------------|
| 飞书 | ✅ webhook 接收 + API 封装 | ❌ 无 `sendFeishuMessage()` |
| 微信 | ⚠️ 仅格式适配器 | ❌ 无 Bot SDK 接入 |
| Web | ✅ REST 适配 | ⚠️ 无 WebSocket 推送 |

**风险**：
- Agent 执行完成后无法回复飞书群聊
- 微信消息无法进入系统
- 定时任务 prompt 无法发送到群聊（当前为模拟执行）

**优化方案**：

## 🟡 低优先级（代码质量/体验优化）

### L1. `next.config.js` 和 `next.config.mjs` 同时存在

`next.config.mjs` 仅 117 字节，疑似残留。Next.js 优先加载 `.mjs`，可能导致 `next.config.js` 中的优化配置不生效。

**方案**：删除 `next.config.mjs`，仅保留 `next.config.js`。

---

### L2. 前端暗色模式类名冗余

多处同时写了重复的 dark mode 类：

```
dark:text-gray-400 dark:text-slate-400
```

后者覆盖前者，`dark:text-gray-400` 永远无效。

**方案**：全局搜索 `dark:text-gray-400 dark:text-slate-400`，统一为一个。

---

### L3. 导入编排器步骤与文档不一致

文档 `项目导入模块.md` 定义了 **11 步**导入流程，但 `importOrchestrator.ts` 只实现了 **8 步**。

缺少的步骤：
- 了解打包机制（step 5）
- 生成功能定位（step 7）
- 历史改动分析（step 11）

**方案**：代码中补齐缺失的步骤，或更新文档与代码对齐。

---

### L4. `lib/api/types.ts` 应拆分为类型和常量

文件名暗示只有类型定义，但实际包含配色映射、价格配置等运行时常量。

**方案**：拆为 `types.ts`（纯 TS 类型）和 `constants.ts`（运行时常量和配置）。

---

### L5. 测试覆盖不均衡

| 已覆盖 | 未覆盖或覆盖不足 |
|--------|-----------------|
| semver、roles、permissions、priorityCalculator | task 路由、message 路由、version 路由、项目导入流程 |

核心业务路由几乎没有集成测试。

**方案**：优先为 task、message、version 三大路由补充 API 级集成测试。

---

### L6. `.husky/` 目录为空，Git hooks 未生效

`package.json` 中有 `"prepare": "husky"`，但 `.husky/` 目录下没有 hook 文件（如 `pre-commit`）。`lint-staged` 配置存在但不会被触发。

**方案**：执行 `npx husky add .husky/pre-commit "npx lint-staged"` 生成 hook 文件。

---

### L7. `server/dist/` 构建产物已进入仓库

构建产物（143 items）不应进入版本控制。

**方案**：在 `.gitignore` 中添加 `server/dist/`，然后 `git rm -r --cached server/dist`。

---

### L8. 空目录残留

- `data/` — 空目录，无任何文件
- `coverage/` — 空目录
- `.git/`、`.husky/`、`.next/`、`.vercel/`、`node_modules/` — 均显示 0 items

**方案**：删除无用的空目录（`data/`、`coverage/`），其余在 `.gitignore` 中处理。
