## 1. config.ts — 配置加载与校验

**文件路径**: `server/src/utils/config.ts`

#### 职责

从 `config.json` 或 `config.example.json` 文件加载服务器配置，包含服务器端口、数据库连接、LLM 模型配置、渠道（飞书/微信）配置等。支持配置校验和缺失必填字段检查。

#### 核心类型

```typescript
interface ServerConfig {
  port: number;
  host: string;
  env: string;
}

interface DatabaseConfig {
  postgres: { host: string; port: number; database: string; user: string; password: string };
  redis: { host: string; port: number; password?: string };
  chromadb: { host: string; port: number };
}

interface ModelsConfig {
  light: { provider: string; model: string; api_key: string; max_tokens: number };
  medium: { provider: string; model: string; api_key: string; max_tokens: number };
  strong: { provider: string; model: string; api_key: string; max_tokens: number };
}

interface ChannelsConfig {
  wechat: { enabled: boolean; puppet: string };
  feishu: { enabled: boolean; app_id: string; app_secret: string };
}

interface Config {
  server: ServerConfig;
  database: DatabaseConfig;
  models: ModelsConfig;
  channels: ChannelsConfig;
}
```

#### 核心函数

| 函数                      | 说明                                                                |
| ------------------------- | ------------------------------------------------------------------- |
| `loadConfig(configPath?)` | 加载配置文件（优先 `config.json`，回退 `config.example.json`）      |
| `validateConfig()`        | 校验环境变量（`PORT`、`DATABASE_URL`、`REDIS_URL`、`CHROMADB_URL`） |
| `checkRequiredEnvVars()`  | 检查 `POSTGRES_PASSWORD` 是否设置（仅警告）                         |

#### 配置加载流程

```
config.json 存在？
  ├── 是 → 读取并 validateConfig()，校验必填字段
  │         ├── 通过 → 返回 Config 对象
  │         └── 失败 → console.error + process.exit(1)
  │
  └── 否 → config.example.json 存在？
            ├── 是 → 读取并返回（附带警告）
            └── 否 → console.error + process.exit(1)
```

#### 必填配置字段

| 字段路径                     | 说明                |
| ---------------------------- | ------------------- |
| `server.port`                | 服务端口（1-65535） |
| `database.postgres.host`     | PostgreSQL 主机     |
| `database.postgres.database` | 数据库名            |
| `database.postgres.user`     | 数据库用户名        |
| `database.redis.host`        | Redis 主机          |
| `database.chromadb.host`     | ChromaDB 主机       |
| `models.light.api_key`       | Light 模型 API Key  |
| `models.medium.api_key`      | Medium 模型 API Key |
| `models.strong.api_key`      | Strong 模型 API Key |

#### 配置文件位置

- 用户配置：`{cwd}/config.json`
- 示例配置：`{cwd}/config.example.json`

#### 使用示例

```typescript
import { loadConfig } from './utils/config.js';

const config = loadConfig();
console.log(config.server.port); // 9700
console.log(config.database.postgres.host);
```

## 2. response.ts — 统一响应构建

**文件路径**: `server/src/utils/response.ts`

#### 职责

提供统一的 API 响应格式构建函数（成功/失败）和可抛出的 `ApiError` 类。所有 Express 路由的响应应通过此模块构建，保证前后端接口格式一致性。

#### 响应类型

```typescript
// 成功响应
interface ApiSuccessResponse<T = unknown> {
  success: true;
  code: 200;
  data: T;
  message: string;
  requestId: string;
}

// 失败响应
interface ApiErrorResponse {
  success: false;
  code: number;
  errorCode: string;
  message: string;
  requestId: string;
  timestamp: string;
  path?: string;
  method?: string;
}
```

#### 核心函数

| 函数签名                                             | 说明                         |
| ---------------------------------------------------- | ---------------------------- |
| `success<T>(data: T, requestId?: string)`            | 构建成功响应                 |
| `error(statusCode, message, errorCode?, requestId?)` | 构建失败响应（支持多种重载） |

#### error() 重载

```typescript
// 方式 1：直接指定状态码
error(400, 'Invalid input', 'BAD_REQUEST');

// 方式 2：仅指定错误码（自动推导状态码）
error('BAD_REQUEST', 'Invalid input'); // 状态码自动映射为 400

// 方式 3：省略 errorCode（自动从状态码推导）
error(404, 'Not found'); // errorCode 自动为 'NOT_FOUND'
```

#### 状态码 ↔ 错误码映射

| 状态码 | 推导错误码       |
| ------ | ---------------- |
| 400    | `BAD_REQUEST`    |
| 401    | `UNAUTHORIZED`   |
| 403    | `FORBIDDEN`      |
| 404    | `NOT_FOUND`      |
| 409    | `CONFLICT`       |
| 429    | `RATE_LIMITED`   |
| 500    | `INTERNAL_ERROR` |

#### ApiError 类

```typescript
class ApiError extends Error {
  statusCode: number;
  errorCode: string;
  requestId: string;

  constructor(
    statusCode: number = 500,
    errorCode: string = 'INTERNAL_ERROR',
    message: string = 'Internal server error'
  );
}
```

#### 使用示例

```typescript
import { success, error, ApiError } from '../utils/response.js';
import { asyncHandler } from '../middleware/errorHandler.js';

// 路由中直接返回
router.get(
  '/versions/:id',
  asyncHandler(async (req, res) => {
    const version = await versionRepo.findById(req.params.id);
    if (!version) {
      res.status(404).json(error(404, 'Version not found', 'NOT_FOUND'));
      return;
    }
    res.json(success(version));
  })
);

// 抛出 ApiError（由 unifiedErrorHandler 捕获）
router.post(
  '/versions',
  asyncHandler(async (req, res) => {
    if (req.body.version.startsWith('invalid')) {
      throw new ApiError(400, 'BAD_REQUEST', 'Version name cannot start with invalid');
    }
    const version = await versionRepo.create(req.body);
    res.json(success(version));
  })
);
```

#### 成功响应示例

```json
{
  "success": true,
  "code": 200,
  "data": { "id": "v1.0.0", "title": "版本 1" },
  "message": "ok",
  "requestId": "a1b2c3d4-e5f6-7890-abcd-ef1234567890"
}
```

## 3. jwt.ts — JWT 身份令牌

**文件路径**: `server/src/utils/jwt.ts`

#### 职责

封装 `jsonwebtoken` 库，提供 Token 签发、验证和刷新 Token 生成功能。Token 包含 `userId` 和 `role` 信息，用于 HTTP 请求的身份认证。

#### 核心类型

```typescript
interface JwtPayload {
  userId: string;
  role: string;
  iat?: number; // issued at
  exp?: number; // expiration
}
```

#### 核心函数

| 函数签名                                      | 说明             | 有效期 |
| --------------------------------------------- | ---------------- | ------ |
| `signToken(payload: { userId, role })`        | 签发访问 Token   | 7 天   |
| `signRefreshToken(payload: { userId, role })` | 签发刷新 Token   | 30 天  |
| `verifyToken(token: string)`                  | 验证并解码 Token | —      |

#### 环境变量

| 变量         | 默认值                            | 说明             |
| ------------ | --------------------------------- | ---------------- |
| `JWT_SECRET` | `dev-secret-change-in-production` | 生产环境必须修改 |

#### 使用示例

```typescript
import { signToken, signRefreshToken, verifyToken } from '../utils/jwt.js';

// 登录时签发 Token
const accessToken = signToken({ userId: user.id, role: user.role });
const refreshToken = signRefreshToken({ userId: user.id, role: user.role });

// 验证 Token（异常时自动抛出 JsonWebTokenError）
const payload = verifyToken(req.headers.authorization!.slice(7));
console.log(payload.userId); // 'ou_xxx'
console.log(payload.role); // 'admin'

// 在 auth.ts 中使用
try {
  const payload = verifyToken(token);
  req.user = { id: payload.userId, role: payload.role as Role };
} catch {
  res.status(401).json(error(401, 'Token invalid or expired'));
}
```

#### 注意事项

- Token 过期自动抛出 `JsonWebTokenError`，调用方需捕获并返回 401
- 生产环境必须设置非默认的 `JWT_SECRET`
- 刷新 Token 用于获取新的访问 Token（当前刷新机制未在代码中实现路由，可自行扩展）

## 4. db.ts — PostgreSQL 连接池

**文件路径**: `server/src/utils/db.ts`

#### 职责

封装 `pg`（node-postgres）的 `Pool`，提供全局复用的数据库连接池，自动注册优雅关闭，在进程退出时正确关闭所有数据库连接。

#### 核心导出

| 导出项 | 类型      | 说明             |
| ------ | --------- | ---------------- |
| `pool` | `pg.Pool` | 全局数据库连接池 |

#### 连接池配置

```typescript
new Pool({
  connectionString: process.env.DATABASE_URL || 'postgresql://localhost:5432/teamclaw',
  max: 20, // 最大连接数
  idleTimeoutMillis: 30000, // 空闲 30s 后释放
  connectionTimeoutMillis: 2000, // 连接超时 2s
});
```

#### 优雅关闭

```typescript
pool.on('error', err => {
  console.error('Unexpected error on idle client', err);
});

onShutdown('PostgreSQL', async () => {
  await pool.end();
});
```

#### 使用示例

```typescript
import { pool } from '../utils/db.js';

// 查询
const result = await pool.query('SELECT * FROM versions WHERE id = $1', [id]);
console.log(result.rows[0]);

// 获取单个连接执行事务
const client = await pool.connect();
try {
  await client.query('BEGIN');
  await client.query('INSERT INTO versions VALUES ($1, $2)', [id, title]);
  await client.query('COMMIT');
} catch (err) {
  await client.query('ROLLBACK');
  throw err;
} finally {
  client.release();
}

// 直接使用 pool.query（自动归还连接）
const { rows } = await pool.query('SELECT COUNT(*) FROM tasks WHERE status = $1', ['completed']);
```

#### 注意事项

- `pool.query()` 自动管理连接生命周期（获取→使用→归还），适合大多数场景
- 事务操作需使用 `pool.connect()` 获取独立连接，用完后 `client.release()`
- 生产环境务必设置 `DATABASE_URL` 环境变量

## 5. redis.ts — Redis 客户端

**文件路径**: `server/src/utils/redis.ts`

#### 职责

封装 `ioredis`，提供全局复用的 Redis 客户端，支持速率限制、会话缓存、发布订阅等场景。自动注册优雅关闭。

#### 核心导出

| 导出项  | 类型    | 说明              |
| ------- | ------- | ----------------- |
| `redis` | `Redis` | 全局 Redis 客户端 |

#### 连接配置

```typescript
new Redis(process.env.REDIS_URL || 'redis://localhost:6379', {
  maxRetriesPerRequest: 3,
});
```

#### 优雅关闭

```typescript
redis.on('error', err => {
  console.error('Redis error:', err);
});

onShutdown('Redis', async () => {
  redis.disconnect();
});
```

#### 使用示例

```typescript
import { redis } from '../utils/redis.js';

// 字符串操作
await redis.set('user:session:123', JSON.stringify({ userId: 'ou_xxx', role: 'admin' }));
const session = JSON.parse((await redis.get('user:session:123')) || '{}');

// 带过期时间
await redis.set('rate:limit:192.168.1.1', '1', 'EX', 60); // 60 秒后自动删除

// 计数器（速率限制）
const count = await redis.incr('rate:limit:ip:192.168.1.1');
if (count > 120) {
  throw new ApiError(429, 'RATE_LIMITED', 'Too many requests');
}

// 哈希表
await redis.hset('task:123', { status: 'in_progress', assignee: 'coder' });
const task = await redis.hgetall('task:123');

// 列表（队列）
await redis.lpush('task:queue', JSON.stringify({ taskId: '123', priority: 10 }));
const job = await redis.rpop('task:queue');

// 发布/订阅
const subscriber = redis.duplicate(); // 订阅频道需要独立连接
await subscriber.subscribe('notifications');
subscriber.on('message', (channel, message) => {
  console.log(`[${channel}] ${message}`);
});
```

#### 注意事项

- 订阅频道需要 `redis.duplicate()` 创建独立连接，不能复用同一个连接
- 生产环境务必设置 `REDIS_URL` 环境变量
- `ioredis` 默认重试策略：最多重试 3 次（`maxRetriesPerRequest: 3`）

## 6. shutdown.ts — 优雅关闭

**文件路径**: `server/src/utils/shutdown.ts`

#### 职责

管理进程优雅关闭（收到 `SIGTERM` / `SIGINT` 信号时），按 LIFO（后进先出）顺序执行所有注册的处理函数，确保数据库连接、Redis 连接等资源正确关闭，不丢失数据。

#### 核心函数

| 函数                         | 说明                              |
| ---------------------------- | --------------------------------- |
| `onShutdown(name, fn)`       | 注册关闭处理函数（LIFO 顺序执行） |
| `registerShutdownHandlers()` | 注册进程信号监听（调用一次）      |

#### 执行顺序

```
收到 SIGTERM / SIGINT
       ↓
按注册顺序的**逆序**（LIFO）执行所有 onShutdown()
       ↓
每个 handler 超时 5 秒
       ↓
process.exit(0)
```

#### 已在模块初始化时注册的关闭处理

| 模块       | 关闭操作             |
| ---------- | -------------------- |
| PostgreSQL | `pool.end()`         |
| Redis      | `redis.disconnect()` |

#### 使用示例

```typescript
import { onShutdown, registerShutdownHandlers } from './shutdown.js';
import { pool } from './db.js';
import { redis } from './redis.js';

// 在需要清理的资源模块中注册
onShutdown('EmailService', async () => {
  await emailService.close();
});

onShutdown('FileUploader', async () => {
  await uploader.cleanup();
});

// 在 index.ts 入口调用一次（通常在所有模块加载之后）
registerShutdownHandlers();
```

#### 处理超时机制

每个关闭 handler 有 5 秒超时限制，超时后强制进入下一个：

```typescript
await Promise.race([
  Promise.resolve(fn()), // 实际关闭逻辑
  new Promise((_, reject) => setTimeout(() => reject(new Error('timeout')), 5000)),
]);
```

#### 注意事项

- `registerShutdownHandlers()` 只应调用一次，通常在 `server/index.ts` 入口
- LIFO 顺序：后注册的服务先关闭（先关应用层，再关基础设施层）
- 未捕获异常（`uncaughtException`）也触发优雅关闭；`unhandledRejection` 仅记录不退出

## 7. chromadb.ts — ChromaDB 向量数据库

**文件路径**: `server/src/utils/chromadb.ts`

#### 职责

封装 [ChromaDB](https://www.trychroma.com/) 客户端，提供向量存储和相似性检索能力（用于 RAG 场景：文档向量化后存储，支持后续语义搜索）。

#### 核心函数

| 函数                          | 说明                                  |
| ----------------------------- | ------------------------------------- |
| `createChromaClient()`        | 获取全局 ChromaClient 单例            |
| `getOrCreateCollection(name)` | 获取或创建同名 Collection（向量空间） |

#### 依赖

- 环境变量：`CHROMA_URL`（默认 `http://localhost:8000`）
- npm 包：`chromadb`

#### 使用示例

```typescript
import { createChromaClient, getOrCreateCollection } from '../utils/chromadb.js';
import { ChromaClient, IEmbeddingFunction } from 'chromadb';

// 获取 Collection（相当于表）
const collection = await getOrCreateCollection('project-docs');

// 添加文档向量
await collection.add({
  ids: ['doc-1', 'doc-2'],
  documents: ['这是一个关于用户认证的技术文档...', '这是一个关于权限管理的功能说明...'],
  metadatas: [
    { projectId: 'proj-123', type: 'auth' },
    { projectId: 'proj-123', type: 'permission' },
  ],
});

// 相似性搜索
const results = await collection.query({
  queryTexts: ['如何实现用户权限控制？'],
  nResults: 5,
});

console.log(results.documents); // 最相似的 5 个文档内容
console.log(results.metadatas); // 对应的元数据
console.log(results.distances); // 相似性距离（越小越相似）
```

#### 完整使用流程（RAG 场景）

```typescript
// 1. 初始化
const collection = await getOrCreateCollection('project-knowledge-base');

// 2. 文档向量化存储（项目导入时）
await collection.add({
  ids: chunkIds,
  documents: chunks,
  metadatas: chunks.map(c => ({ source: c.filePath, projectId })),
});

// 3. 语义检索（Agent 查询时）
const { documents, distances } = await collection.query({
  queryTexts: [userQuestion],
  nResults: 3,
});

// 4. 过滤高相似度结果（距离 < 1.0）
const relevant = documents[0].filter((_, i) => distances[0][i] < 1.0);
```

#### 注意事项

- ChromaDB 默认使用 `all-MiniLM-L6-v2` 内置 embedding 函数（无需额外配置 API Key）
- 生产环境务必设置 `CHROMA_URL` 环境变量指向独立的 ChromaDB 服务
- `createChromaClient()` 返回单例，避免重复连接

## 8. config-validator.ts — 环境变量校验

**文件路径**: `server/src/utils/config-validator.ts`

#### 职责

在服务启动时校验所有必需的环境变量是否存在，验证 API Key 格式，提供启动前的配置检查清单。**独立可执行**：直接运行 `node server/dist/utils/config-validator.js` 即可检查当前环境配置。

#### 环境变量分级

| 级别          | 未设置行为         | 说明                           |
| ------------- | ------------------ | ------------------------------ |
| `required`    | **退出（exit 1）** | 缺失将导致服务无法启动         |
| `recommended` | 仅警告             | 缺失不影响启动但功能可能不完整 |

#### required（必填）

| 变量名     | 说明     |
| ---------- | -------- |
| `PORT`     | 服务端口 |
| `NODE_ENV` | 运行环境 |

#### recommended（推荐）

| 变量名             | 说明                |
| ------------------ | ------------------- |
| `DATABASE_URL`     | PostgreSQL 连接地址 |
| `REDIS_URL`        | Redis 连接地址      |
| `JWT_SECRET`       | JWT 签名密钥        |
| `DEEPSEEK_API_KEY` | DeepSeek API Key    |
| `SERVER_URL`       | 服务器公开 URL      |

#### 额外校验规则

| 规则                     | 条件                  | 行为 |
| ------------------------ | --------------------- | ---- |
| `JWT_SECRET` 非默认值    | `NODE_ENV=production` | 退出 |
| `DEEPSEEK_API_KEY` 格式  | 若设置 → 必须以 `sk-` | 退出 |
| `OPENAI_API_KEY` 格式    | 若设置 → 必须以 `sk-` | 退出 |
| `ANTHROPIC_API_KEY` 格式 | 若设置 → 必须以 `sk-` | 退出 |

#### 使用方式

```bash
# 独立运行检查
node server/dist/utils/config-validator.js

# 输出示例（全部通过）
[config-validator] All checks passed!

# 输出示例（有缺失）
[config-validator] Warnings:
  - Missing recommended env var: REDIS_URL
[config-validator] Errors:
  - Missing required env var: PORT
  - DEEPSEEK_API_KEY appears invalid (should start with sk-)
```

#### 在服务入口集成

```typescript
// server/src/index.ts
import { validateConfig } from './utils/config.js';
import { registerShutdownHandlers } from './utils/shutdown.js';

// 启动前强制校验
const { valid, errors } = validateConfig();
if (!valid) {
  console.error('Environment validation failed:', errors);
  process.exit(1);
}

registerShutdownHandlers();
// 启动 Express...
```

#### 注意事项

- 该模块执行后调用 `process.exit(0)` 或 `process.exit(1)`，因此**不应**作为库导入使用，仅独立执行
- 编译后的路径为 `server/dist/utils/config-validator.js`，需先执行 `npm run build`
- API Key 格式校验基于开头字符，属于基础格式校验，不验证 Key 实际有效性

```
config-validator.ts    ← 环境变量检查（独立执行脚本）
     │
     ├── db.ts (PostgreSQL)
     │      └── pool.on('error') → onShutdown('PostgreSQL', pool.end)
     │
     ├── redis.ts (Redis)
     │      └── redis.on('error') → onShutdown('Redis', redis.disconnect)
     │
     ├── chromadb.ts (ChromaDB)
     │      └── createChromaClient()
     │
     └── shutdown.ts (优雅关闭)
            ├── registerShutdownHandlers() → SIGTERM / SIGINT / uncaughtException
            └── LIFO 执行所有 onShutdown() 注册的清理函数
```
