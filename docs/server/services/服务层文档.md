## 1. Git 操作

### 1.1 gitClone.ts — 项目克隆

**文件路径**: `server/src/services/gitClone.ts`

#### 职责

将远程 Git 仓库或本地已有项目目录接入系统，是项目导入流程的第一步（`clone` 阶段）。提供安全校验，防止路径遍历和 shell 注入。

#### 核心函数

```typescript
cloneOrCopyProject(
  source: 'url' | 'local',
  url?: string,
  localPath?: string
): Promise<string>
```

| 参数        | 类型               | 说明                                  |
| ----------- | ------------------ | ------------------------------------- |
| `source`    | `'url' \| 'local'` | 来源类型                              |
| `url`       | `string`           | Git 远程地址（source='url' 时必填）   |
| `localPath` | `string`           | 本地绝对路径（source='local' 时必填） |

**返回值**: 项目本地路径（`~/.openclaw/projects/{repoName}` 或原路径）

#### 安全校验

| 校验项       | 实现                                                  |
| ------------ | ----------------------------------------------------- |
| URL 白名单   | 仅允许 `http://`、`https://`、`git@` 开头             |
| 非法字符拦截 | 禁止 `; \| & $ \` > < ( ) { }` 等 shell 元字符        |
| 路径遍历防护 | 禁止 `..` 路径组件                                    |
| 系统目录保护 | 禁止访问 `/etc /usr /bin /sbin /var /root /sys /proc` |

#### 使用示例

```typescript
import { cloneOrCopyProject } from './gitClone.js';

// 从远程 URL 克隆
const projectPath = await cloneOrCopyProject('url', 'https://github.com/user/repo.git');

// 从本地上传的项目目录接入
const projectPath = await cloneOrCopyProject('local', undefined, '/Users/me/projects/myapp');
```

#### 注意事项

- 使用 `execFile`（非 shell）执行 git 命令，彻底避免 shell 注入
- 仓库名白名单：`/^[\w\-_.]+$/`
- 已存在的目录执行 `git pull` 而非重新 clone（`--depth 1` 浅克隆）

### 1.2 gitService.ts — Git 元数据读写

**文件路径**: `server/src/services/gitService.ts`

#### 职责

对已接入的项目仓库进行只读 Git 操作，提取版本历史、标签、分支、提交记录等元数据，供版本管理、变更追踪、历史分析等模块使用。

#### 核心类型

```typescript
interface GitCommit {
  hash: string; // 完整 SHA
  shortHash: string; // 短 SHA
  message: string; // 提交信息
  author: string; // 作者名
  authorEmail: string; // 作者邮箱
  date: string; // ISO 日期
  filesChanged?: number;
  insertions?: number;
  deletions?: number;
}

interface GitTag {
  name: string;
  commit: string;
  date: string;
  message?: string;
}

interface GitBranch {
  name: string;
  isCurrent: boolean;
  isRemote: boolean;
}
```

#### 核心函数

| 函数                                       | 说明                       | 返回值                           |
| ------------------------------------------ | -------------------------- | -------------------------------- |
| `getGitLog(cwd, options)`                  | 获取提交历史               | `GitCommit[]`                    |
| `getTags(cwd)`                             | 获取所有标签               | `GitTag[]`                       |
| `createTag(cwd, tagName, message?)`        | 创建标签                   | `boolean`                        |
| `deleteTag(cwd, tagName)`                  | 删除标签                   | `boolean`                        |
| `getBranches(cwd)`                         | 获取分支列表               | `GitBranch[]`                    |
| `getCurrentBranch(cwd)`                    | 获取当前分支名             | `string`                         |
| `createBranch(cwd, branchName, baseRef?)`  | 创建分支                   | `boolean`                        |
| `checkout(cwd, ref)`                       | 切换到指定引用             | `boolean`                        |
| `getDiffStats(cwd, fromRef, toRef)`        | 两引用间差异统计           | `{files, insertions, deletions}` |
| `tagExists(cwd, tagName)`                  | 标签是否存在               | `boolean`                        |
| `getTagCommit(cwd, tagName)`               | 获取标签对应 commit        | `string`                         |
| `getTagDetails(cwd, tagName)`              | 标签详细信息（含注解作者） | `TagDetails \| null`             |
| `getCommitsBetween(cwd, fromRef, toRef)`   | 两引用间的提交列表         | `GitCommit[]`                    |
| `compareBranches(cwd, branch, baseBranch)` | 分支领先/落后 commit 数    | `{ahead, behind}`                |

#### 使用示例

```typescript
import { getGitLog, getTags, getDiffStats, compareBranches } from './gitService.js';

const commits = getGitLog('/path/to/repo', { maxCount: 10, since: '2026-01-01' });
const tags = getTags('/path/to/repo');
const diff = getDiffStats('/path/to/repo', 'v1.0.0', 'v1.1.0');
const comparison = compareBranches('/path/to/repo', 'feature-x', 'main');
```

#### 注意事项

- 所有操作均为同步（`execFileSync`），适用于 CLI/路由层
- 自动跳过非 Git 目录（`.git` 不存在时返回空值/false）
- 超时时间 30 秒

## 2. 项目分析

### 2.1 fileScanner.ts — 文件结构扫描

**文件路径**: `server/src/services/fileScanner.ts`

#### 职责

递归扫描项目目录结构，生成目录树（用于 UI 展示项目结构）。自动忽略 `node_modules`、`.git`、二进制文件等无分析价值路径。

#### 核心类型

```typescript
interface DirNode {
  name: string;
  path: string;
  type: 'file' | 'directory';
  children?: DirNode[]; // 仅 directory 有
  size?: number; // 仅 file 有（字节）
  extension?: string; // 仅 file 有
}
```

#### 核心函数

```typescript
scanDirectory(rootPath: string, maxDepth = 10): Promise<DirNode>
```

#### 忽略规则

| 类型       | 规则                                                                                |
| ---------- | ----------------------------------------------------------------------------------- |
| 目录       | `node_modules`, `.git`, `dist`, `build`, `.next`, `.cache`, `__pycache__`           |
| 文件扩展名 | `.lock`, `.log`, `.ico`, `.png`, `.jpg`, `.jpeg`, `.gif`, `.svg`, `.woff`, `.woff2` |

#### 使用示例

```typescript
import { scanDirectory } from './fileScanner.js';

const tree = await scanDirectory('/path/to/project');
// tree.children[0].name === 'src'
// tree.children[0].children => [DirNode, DirNode, ...]
```

#### 注意事项

- 异步递归，`maxDepth` 防止无限递归（默认 10 层）
- 目录先于文件排序，同类型按名称字母排序

### 2.2 techDetector.ts — 技术栈检测

**文件路径**: `server/src/services/techDetector.ts`

#### 职责

通过扫描项目根目录及子目录文件，检测项目使用的技术栈（语言、框架、构建工具、ORM、运行时等），为项目导入提供技术上下文。

#### 核心类型

```typescript
interface TechStack {
  framework: string[]; // Next.js / Vue / React / Flutter ...
  language: string[]; // TypeScript / Python / Go / Rust ...
  buildTool: string[]; // npm / Cargo / Gradle / CMake ...
  orm: string[]; // Prisma / Sequelize / Hibernate ...
  uiFramework: string[]; // 同 framework（冗余字段，兼容旧接口）
  runtime: string[]; // Node.js / Bun / Deno / JVM ...
  fullStack: string[]; // SSR / API Layer / Backend API ...
}
```

#### 检测逻辑

| 检测维度 | 检测方式                                                                  |
| -------- | ------------------------------------------------------------------------- |
| 框架     | 根目录文件名匹配：`next.config.js` → Next.js，`angular.json` → Angular 等 |
| 语言     | 文件扩展名统计：`.ts` → TypeScript，`.py` → Python                        |
| 构建工具 | 根目录配置文件：`package.json` → npm/yarn/pnpm，`Cargo.toml` → Cargo      |
| ORM      | `package.json` dependencies 解析 + 模糊匹配                               |
| 运行时   | 配置文件推断：go.mod → Go，pyproject.toml → Python                        |
| 全栈     | 目录结构 + 框架特征：server/ 或 api/ → API Layer                          |

#### 使用示例

```typescript
import { detectTechStack } from './techDetector.js';

const tech = await detectTechStack('/path/to/project');
console.log(tech.framework); // ['Next.js', 'React']
console.log(tech.language); // ['TypeScript', 'TypeScript/React']
console.log(tech.buildTool); // ['npm/yarn/pnpm']
console.log(tech.orm); // ['Prisma']
```

## 3. 项目导入编排

### 3.1 importOrchestrator.ts — 导入流程编排器

**文件路径**: `server/src/services/importOrchestrator.ts`

#### 职责

管理项目从接入到完成的全流程多步骤编排，是 **H5 导入流程异步串联**（Issue #375 关联）的核心实现。每个步骤对应一个独立的 Service 调用，步骤之间通过 `ImportContext` 传递上下文，支持断点续传（从 PostgreSQL 恢复）和部分失败容错。

#### 导入步骤（ALL_STEPS）

按执行顺序排列：

| #   | 步骤名               | 说明                  | 关键依赖                               |
| --- | -------------------- | --------------------- | -------------------------------------- |
| 1   | `clone`              | 克隆或定位项目        | `gitClone.cloneOrCopyProject`          |
| 2   | `scan`               | 扫描文件结构          | `fileScanner.scanDirectory`            |
| 3   | `detectTech`         | 检测技术栈            | `techDetector.detectTechStack`         |
| 4   | `parseDocs`          | 解析文档（README 等） | `docParser.parseDirectory`             |
| 5   | `analyzeCode`        | 分析代码架构          | `codeAnalyzer.analyzeCodeArchitecture` |
| 6   | `detectBuild`        | 检测打包机制          | `buildDetector.detectBuildMechanism`   |
| 7   | `compress`           | 上下文压缩            | `contextCompressor.chunkProject`       |
| 8   | `buildSummary`       | 生成项目摘要          | `summaryGenerator.generateSummary`     |
| 9   | `generateFeatureMap` | 生成功能定位          | `featureMap.generateFeatureMap`        |
| 10  | `generateSkills`     | 生成 Skills           | `skillGenerator.generateSkills`        |
| 11  | `convertDocs`        | 文档格式转换          | `docConverter.convertDocuments`        |
| 12  | `vectorize`          | 向量化存储            | `vectorStore.addDocuments`             |
| 13  | `gitHistory`         | Git 历史分析          | `gitHistoryAnalysis.analyzeGitHistory` |
| 14  | `done`               | 终端步骤（空操作）    | —                                      |

#### 关键类型

```typescript
interface ImportContext {
  source: 'url' | 'local';
  url?: string;
  localPath?: string;
  projectPath?: string;
  projectId?: string;
  projectName?: string;
  techStack?: TechStack;
  tree?: DirNode;
  documents?: ParsedDocument[];
  architecture?: CodeArchitecture;
  buildMechanisms?: BuildMechanism[];
  summary?: ProjectSummary;
  featureMap?: FeatureMap;
  skills?: GeneratedSkill[];
  convertedDocs?: ConvertedDoc[];
  vectorCollectionName?: string;
  chunks?: TextChunk[];
}

type ImportStepName = /* 见上表 */;
type ImportTaskStatus = 'pending' | 'processing' | 'done' | 'error';
```

#### 核心 API

| 函数                                                  | 说明                                                     |
| ----------------------------------------------------- | -------------------------------------------------------- |
| `createImportTask(projectId)`                         | 创建导入任务，返回 `ImportTask` 对象                     |
| `runAllSteps(taskId, ctx)`                            | **异步**执行所有步骤（非阻塞，用于 HTTP 响应后继续执行） |
| `getTaskStatus(taskId)`                               | 查询任务当前状态                                         |
| `executeStep(taskId, stepIndex)`                      | 标记步骤为 running（内部调用）                           |
| `completeStep(taskId, stepIndex, error?, isWarning?)` | 标记步骤完成（含错误处理）                               |
| `stepDisplayName(step)`                               | 获取步骤中文显示名                                       |

#### 非关键步骤容错

以下步骤失败仅打印警告，不阻塞后续步骤：

```
vectorize | generateSkills | generateFeatureMap | convertDocs | gitHistory
```

#### 使用示例

```typescript
import { createImportTask, runAllSteps, getTaskStatus } from './importOrchestrator.js';

const task = createImportTask(projectId);

const ctx: ImportContext = {
  source: 'url',
  url: 'https://github.com/user/repo.git',
  projectId,
  projectName: 'my-project',
};

runAllSteps(task.taskId, ctx).catch(console.error);

// 后续轮询状态
const status = getTaskStatus(task.taskId);
console.log(status.steps[status.currentStep - 1]); // 当前步骤
```

#### PostgreSQL 持久化

任务状态通过 `importRepo.upsert()` 实时写入 PostgreSQL，服务重启后可从 `import_repo` 表恢复 `pending` 状态的任务（`loadFromDb()` 在模块初始化时自动调用）。

## 4. Agent 执行引擎

### 4.1 agentExecution.ts — Agent 任务执行

**文件路径**: `server/src/services/agentExecution.ts`

#### 职责

负责实际触发 Agent（main/pm/coder/reviewer）执行任务，是 Agent 派发的执行层核心。处理权限校验、Agent 状态管理、LLM 调用编排、执行超时控制和结果回传。

#### 核心类型

```typescript
interface ExecutionContext {
  executionId: string; // exec_${timestamp}_${random}
  taskId: string;
  dispatcher: string; // 发起者
  targetAgent: string; // 执行者
  prompt: string;
  createdAt: string;
  startedAt?: string;
  completedAt?: string;
  status: 'pending' | 'running' | 'completed' | 'failed' | 'timeout';
  result?: string; // LLM 输出内容
  error?: string;
  model?: string;
  durationMs?: number;
  usage?: {
    inputTokens: number;
    outputTokens: number;
    totalTokens: number;
  };
  costUsd?: number;
}

interface DispatchRequest {
  dispatcher: string; // 'system' | 'main' | 'pm' ...
  targetAgent: string; // 'main' | 'pm' | 'coder' | 'reviewer'
  taskId: string;
  prompt: string;
  timeoutMs?: number; // 默认 5 分钟
  model?: string;
}
```

#### 核心 API

| 函数                                 | 说明                                                          |
| ------------------------------------ | ------------------------------------------------------------- |
| `dispatchToAgent(req)`               | **同步**派发任务，立即返回 `ExecutionContext`（实际执行异步） |
| `getExecution(executionId)`          | 查询执行记录                                                  |
| `getAgentExecutionState(agentName)`  | 获取 Agent 当前执行状态                                       |
| `getExecutionHistory(opts)`          | 分页查询执行历史                                              |
| `abortExecution(agentName, reason)`  | 强制终止 Agent 当前执行                                       |
| `getAgentExecutionStats(agentName?)` | 获取统计信息（总数/完成/失败/超时/平均耗时）                  |

#### 执行流程

```
dispatchToAgent() 同步返回
  ↓
executeAgentTask() 异步执行
  ├── buildAgentMessages()  → 构建 system + user prompt
  ├── llmAutoRoute()        → 自动路由选择模型
  ├── llmCostTracker.record() → 记录 token 成本
  ├── taskMemory.addMessage()  → 持久化对话历史
  └── updateExecution()     → 写入执行结果
       ↓
onExecutionComplete() 异步回调
  ├── sendReply() → 回复消息通道
  └── maybeDispatchSubtasks() → [DISPATCH: agent: desc] 格式自动派发子任务
```

#### 子任务自动派发

Agent 响应中包含 `[DISPATCH: agentName: taskDescription]` 格式指令时，自动派发子任务（最大递归深度 3 层，防止循环）：

```
[DISPATCH: coder: 修复首页样式问题]
[DISPATCH: pm: 评审技术方案]
```

#### 权限校验

通过 `canDispatch(dispatcher, targetAgent)` 检查派发权限（定义在 `constants/agents.ts`）。

#### 使用示例

```typescript
import { dispatchToAgent, getExecution, getAgentExecutionStats } from './agentExecution.js';

// 派发任务
const result = dispatchToAgent({
  dispatcher: 'main',
  targetAgent: 'coder',
  taskId: 'task_123',
  prompt: '请修复首页的登录按钮样式',
  timeoutMs: 5 * 60 * 1000,
});

if ('error' in result) {
  console.error('派发失败:', result.error);
} else {
  console.log('executionId:', result.executionId);
  // 异步执行，结果通过 getExecution() 查询
}

// 统计
const stats = getAgentExecutionStats('coder');
console.log(stats.coder.completed, stats.coder.avgDurationMs);
```

## 5. LLM 服务

### 5.1 llmService.ts — 多模型统一调用

**文件路径**: `server/src/services/llmService.ts`

#### 职责

封装 DeepSeek / OpenAI / Anthropic 三家 API，提供统一接口。按任务复杂度自动路由选择模型层级（light/medium/strong），支持降级失败重试和成本追踪。

#### 模型层级

| 层级     | 模型                       | 提供商    | 最大 Token | 典型场景                     |
| -------- | -------------------------- | --------- | ---------- | ---------------------------- |
| `light`  | `deepseek-chat`            | DeepSeek  | 4096       | 简单问答、格式生成           |
| `medium` | `gpt-4o-mini`              | OpenAI    | 16384      | 中等复杂度任务               |
| `strong` | `claude-sonnet-4-20250514` | Anthropic | 81920      | 复杂分析、代码生成、架构设计 |

环境变量配置：

```bash
LIGHT_MODEL=deepseek-chat      # 默认
MEDIUM_MODEL=gpt-4o-mini       # 默认
STRONG_MODEL=claude-sonnet-4-20250514  # 默认
DEEPSEEK_API_KEY=...
OPENAI_API_KEY=...
ANTHROPIC_API_KEY=...
```

#### 核心 API

| 函数                                                | 说明                                            |
| --------------------------------------------------- | ----------------------------------------------- |
| `llmCall(request, fallbackTiers?)`                  | 统一入口，支持多 tier 降级                      |
| `llmCallLight(messages, fallback?)`                 | 快捷方法：light tier                            |
| `llmCallMedium(messages, fallback?)`                | 快捷方法：medium tier                           |
| `llmCallStrong(messages)`                           | 快捷方法：strong tier                           |
| `llmAutoRoute(messages, overrideTier?)`             | **自动路由**：根据内容复杂度选择 tier           |
| `estimateTokens(text)`                              | 估算 token 数量（中文/英文/代码混合）           |
| `estimateMessageTokens(messages)`                   | 估算消息列表总 token                            |
| `estimateCost(provider, inputTokens, outputTokens)` | 成本估算（基于参考定价）                        |
| `estimateComplexity(text)`                          | 任务复杂度判断（`simple \| medium \| complex`） |
| `selectTierByComplexity(complexity)`                | 复杂度 → 模型层级映射                           |

#### 自动路由规则

```typescript
estimateComplexity() 规则：
- 简单：文本 < 200 字符，无代码块，无架构关键词
- 复杂：文本 > 2000 字符 或 代码块 > 3 或 包含"架构/设计/review/重构"
- 中等：其余情况
```

#### 使用示例

```typescript
import { llmAutoRoute, llmCallStrong, estimateTokens } from './llmService.js';

// 自动路由（推荐）
const response = await llmAutoRoute([
  { role: 'system', content: '你是一个代码审查助手' },
  { role: 'user', content: '审查以下代码...' },
]);

console.log(response.content); // LLM 回复
console.log(response.provider); // 'anthropic' | 'openai' | 'deepseek'
console.log(response.usage); // { inputTokens, outputTokens, totalTokens }

// 直接指定 strong tier
const result = await llmCallStrong([{ role: 'user', content: '用 TypeScript 实现一个 LRU Cache' }]);

// 估算成本
const cost = estimateCost('anthropic', 1000, 500);
console.log(`预估成本: $${cost.toFixed(4)}`);
```

#### 降级机制

```typescript
// llmAutoRoute 默认降级链
simple  → light 失败 → medium → strong
medium  → medium 失败 → strong
complex → 仅 strong（不降级）
```

429（限流）或 5xx 错误触发降级；401/403 等不可重试错误立即中断。

## 6. 消息→任务→Agent 流水线

### 6.1 messagePipeline.ts — 消息主管道

**文件路径**: `server/src/services/messagePipeline.ts`

#### 职责

消息进入系统的统一入口，串联消息接收→权限校验→任务创建→Agent 派发全流程。通过 EventBus 驱动下游的 `messageToTask` 和 `taskToAgent` 模块。

#### 核心流程

```
接收消息 (ReceiveMessageRequest)
  ↓
1. 入队 messageQueueService（计算优先级）
  ↓
2. 立即发送「消息已收到」回复（异步，不阻塞）
  ↓
3. emit('message:routed')
  ↓
  ├─ 无 @agent → 仅入队，messageToTask 创建普通任务
  │
  └─ 有 @agent → handleAgentMention()
       ├─ 权限校验 (checkPermission)
       ├─ 任务创建 (taskLifecycle.createTask)
       ├─ Agent 派发 (dispatchToAgent)
       ├─ 任务状态 → running
       └─ emit('task:created')
```

#### @Agent 权限矩阵（简化）

| 用户角色 | @main | @pm | @coder | @reviewer |
| -------- | ----- | --- | ------ | --------- |
| 管理员   | ✅    | ✅  | ✅     | ✅        |
| 副管理员 | ✅    | ✅  | ✅     | ❌        |
| 普通员工 | ✅    | ❌  | ❌     | ❌        |

#### 核心 API

```typescript
processMessage(request: ReceiveMessageRequest): Promise<PipelineResult>

interface PipelineResult {
  success: boolean;
  messageId: string;
  acknowledged?: boolean;
  taskId?: string;
  executionId?: string;
  error?: string;
}
```

#### 使用示例

```typescript
import { processMessage } from './messagePipeline.js';

const result = await processMessage({
  channel: 'feishu',
  userId: 'ou_xxx',
  userName: '张三',
  role: 'admin',
  content: '@coder 帮我修复首页 Bug',
  mentionedAgent: 'coder',
  timestamp: new Date().toISOString(),
});
// result.executionId === 'exec_xxx'
```

#### @Agent 检测

```typescript
detectMentionedAgent(content: string): AgentName | null
// 支持：@main / @pm / @coder / @reviewer（不区分大小写）
```

### 6.2 messageToTask.ts — 消息→任务转换

**文件路径**: `server/src/services/messageToTask.ts`

#### 职责

监听 `message:routed` 事件，自动将消息转换为任务，并注入历史上下文。是消息管道和任务管道之间的桥梁。

#### 触发逻辑

```
eventBus.on('message:routed', ...)
  │
  ├─ skipTaskCreation=true（已有 @agent 任务）→ 跳过创建，从 store 查找最近任务
  │
  └─ skipTaskCreation=false → taskLifecycle.createTask() 新建任务
       │
       └─ taskMemory.enrichTaskContext() → 注入历史相关任务摘要
            │
            └─ emit('task:created')
```

#### 优先级映射

```
消息优先级 ≥ 20  →  'urgent'
消息优先级 ≥ 15  →  'high'
消息优先级 ≥ 8   →  'normal'
其他            →  'low'
```

#### 注意事项

- `skipTaskCreation=true` 路径下，通过 `createdBy + createdAt` 查找最近创建的任务（依赖创建顺序）
- 历史上下文注入失败不影响主流程（`try/catch` 包装）

---

### 6.3 taskToAgent.ts — 任务→Agent 触发

**文件路径**: `server/src/services/taskToAgent.ts`

#### 职责

监听 `task:created` 事件，自动触发 Agent 协作流水线（`agentPipeline`），是任务到 Agent 执行的关键衔接点。

#### 流程

```
eventBus.on('task:created', ...)
  │
  ├─ taskLifecycle.transition(taskId, 'running')
  ├─ emit('agent:pipeline:start')
  ├─ createPipeline(taskId, title)    → agentPipeline.ts
  ├─ executePipeline(pipeline.id)     → 实际运行 Agent 协作
  ├─ emit('agent:pipeline:done')
  │
  └─ 失败时 emit('task:failed')
```

#### 注意事项

- 整个 `task:created` 处理是**异步**的，不阻塞 HTTP 响应
- `agentPipeline.ts` 负责多 Agent 协作编排（main → pm/coder/reviewer 等）
