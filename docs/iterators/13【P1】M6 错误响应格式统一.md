# 【P1】M6 错误响应格式统一

> 优先级：P1（中）
> 前置依赖：无 · 后续影响：前后端所有 API 交互的错误处理

---

## 1. 问题描述

项目中存在**两种不兼容的错误响应格式**：

### 格式 A（errorHandler 中间件）

```json
{
  "error": true,
  "code": "NOT_FOUND",
  "message": "Route GET /xxx not found",
  "requestId": "uuid-xxx",
  "timestamp": "2026-03-23T15:00:00.000Z",
  "path": "/xxx",
  "method": "GET"
}
```

来源：`server/src/middleware/errorHandler.ts` 中的 `unifiedErrorHandler` 和 `notFoundHandler`。

### 格式 B（路由中直接返回）

```json
{
  "code": 401,
  "data": null,
  "message": "未提供身份信息"
}
```

来源：`auth.ts` 中间件、各路由的 `res.json(error(400, '...'))` 调用。

### 问题影响

| 场景 | 前端需要处理 |
|------|------------|
| auth 中间件拦截 | 格式 B：`res.code === 401` |
| 路由中业务错误 | 格式 B：`res.code === 400` |
| 未捕获异常 | 格式 A：`res.error === true && res.code === 'INTERNAL_ERROR'` |
| 404 | 格式 A：`res.error === true && res.code === 'NOT_FOUND'` |

前端 `lib/api/` 中的 fetch 函数需要**同时处理两种结构**，代码冗余且易出错。

## 2. 当前代码分析

### errorHandler.ts 的格式

```typescript
// server/src/middleware/errorHandler.ts
interface ErrorResponse {
  error: true;
  code: string;        // "NOT_FOUND", "INTERNAL_ERROR" 等字符串
  message: string;
  requestId: string;
  timestamp: string;
  path?: string;
  method?: string;
}
```

### response.ts 的工具函数

```typescript
// server/src/utils/response.ts
export function success(data: any) {
  return { code: 200, data, message: 'ok' };
}
export function error(code: number, message: string) {
  return { code, data: null, message };   // code 是数字！
}
```

### auth.ts 中的错误

```typescript
res.status(401).json({
  code: 401,           // 数字
  data: null,
  message: "未提供身份信息",
});
```

### 冲突点总结

| 字段 | 格式 A | 格式 B |
|------|--------|--------|
| `error` | `true`（存在） | 不存在 |
| `code` | 字符串（`"NOT_FOUND"`） | 数字（`401`） |
| `data` | 不存在 | `null` |
| `requestId` | 有 | 无 |
| `timestamp` | 有 | 无 |

---

## 3. 统一方案

### 目标格式

统一为格式 A 的增强版，兼具两者的优点：

```typescript
interface ApiResponse<T = unknown> {
  // 成功响应
  success: true;
  code: 200;
  data: T;
  message: string;
  requestId: string;
}

interface ApiErrorResponse {
  // 错误响应
  success: false;
  code: number;            // HTTP 状态码（数字，兼容格式 B）
  errorCode: string;       // 业务错误码（字符串，兼容格式 A）
  message: string;
  requestId: string;
  timestamp: string;
  path?: string;
  method?: string;
}
```

## 4. 实现步骤

### Step 1：重写 `response.ts`（0.5h）

```typescript
// server/src/utils/response.ts
import { randomUUID } from 'crypto';

export interface ApiSuccessResponse<T = unknown> {
  success: true;
  code: 200;
  data: T;
  message: string;
  requestId: string;
}

export interface ApiErrorResponse {
  success: false;
  code: number;
  errorCode: string;
  message: string;
  requestId: string;
  timestamp: string;
}

export function success<T>(data: T, requestId?: string): ApiSuccessResponse<T> {
  return {
    success: true,
    code: 200,
    data,
    message: 'ok',
    requestId: requestId || randomUUID(),
  };
}

export function error(
  statusCode: number,
  message: string,
  errorCode?: string,
  requestId?: string
): ApiErrorResponse {
  return {
    success: false,
    code: statusCode,
    errorCode: errorCode || statusCodeToErrorCode(statusCode),
    message,
    requestId: requestId || randomUUID(),
    timestamp: new Date().toISOString(),
  };
}

function statusCodeToErrorCode(status: number): string {
  const map: Record<number, string> = {
    400: 'BAD_REQUEST',
    401: 'UNAUTHORIZED',
    403: 'FORBIDDEN',
    404: 'NOT_FOUND',
    429: 'RATE_LIMITED',
    500: 'INTERNAL_ERROR',
  };
  return map[status] || 'UNKNOWN_ERROR';
}
```

### Step 2：更新 errorHandler.ts（0.5h）

统一 `unifiedErrorHandler` 和 `notFoundHandler` 使用新的 `error()` 函数：

```typescript
// server/src/middleware/errorHandler.ts
import { error as errorResponse, ApiErrorResponse } from '../utils/response.js';

export function unifiedErrorHandler(
  err: Error | ApiError,
  req: Request,
  res: Response,
  _next: NextFunction
): void {
  let statusCode = 500;
  let errorCode = 'INTERNAL_ERROR';
  let message = 'Internal server error';

  if (err instanceof ApiError) {
    statusCode = err.statusCode;
    errorCode = err.errorCode;
    message = err.message;
  }
  // ... 其他判断逻辑保持

  const body = errorResponse(statusCode, message, errorCode);
  // 补充请求信息
  res.status(statusCode).json({ ...body, path: req.path, method: req.method });
}
```

### Step 3：更新 auth.ts 中间件（0.5h）

所有中间件改用 `error()` 函数：

```typescript
// Before
res.status(401).json({ code: 401, data: null, message: "未提供身份信息" });

// After
import { error } from '../utils/response.js';
res.status(401).json(error(401, '未提供身份信息', 'UNAUTHORIZED'));
```

需要修改的函数：
- `requirePermission()` — 2 处
- `requireAuth()` — 1 处
- `requireAdmin()` — 2 处

### Step 4：更新所有路由中的错误返回（2h）

全局搜索并替换：

```bash
# 查找所有直接构造错误响应的地方
grep -rn "res\.status.*json.*code:" server/src/routes/ --include="*.ts"
grep -rn "res\.json(error(" server/src/routes/ --include="*.ts"
```

需要确保所有 `res.json(error(400, '...'))` 的调用格式一致。

由于 `error()` 函数签名变了（增加了 `errorCode` 参数），需要检查所有调用点：

```typescript
// 原调用
res.status(400).json(error(400, 'Missing field'));

// 新调用（向后兼容，errorCode 可选）
res.status(400).json(error(400, 'Missing field'));  // 自动推导 errorCode = 'BAD_REQUEST'
// 或显式指定
res.status(400).json(error(400, 'Missing field', 'VALIDATION_ERROR'));
```

### Step 5：更新前端错误解析（1h）

在 `lib/api-shared.ts` 中统一错误处理：

```typescript
export interface ApiError {
  success: false;
  code: number;
  errorCode: string;
  message: string;
  requestId: string;
}

export function isApiError(data: unknown): data is ApiError {
  return typeof data === 'object' && data !== null && (data as any).success === false;
}

export async function handleResponse<T>(res: Response): Promise<T> {
  const json = await res.json();

  if (!res.ok || json.success === false) {
    const err = new Error(json.message || `HTTP ${res.status}`);
    (err as any).code = json.code;
    (err as any).errorCode = json.errorCode;
    (err as any).requestId = json.requestId;
    throw err;
  }

  return json.data as T;
}
```

## 6. 验收标准

| # | 验收项 | 验证方式 |
|---|--------|---------|
| 1 | 所有 4xx/5xx 响应包含 `success: false` | `curl` 各种错误场景 |
| 2 | 所有错误响应包含 `requestId` | 检查返回 JSON |
| 3 | 所有错误响应包含 `errorCode`（字符串） | 检查返回 JSON |
| 4 | 所有成功响应包含 `success: true` | `curl` 正常请求 |
| 5 | 前端能用统一的 `isApiError()` 判断错误 | 前端代码审查 |
| 6 | 代码中无 `{ code: 401, data: null, message }` 格式 | `grep -rn "data: null" server/src/` |
| 7 | 现有测试通过 | `npm test` |
