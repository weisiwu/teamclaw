# 【P0】H5 导入流程异步编排串联

> 优先级：P0（高）
> 前置依赖：无（可独立执行，但完成 H1 数据存储统一后可持久化任务状态）
> 关联模块：[项目导入模块](../modules/项目导入模块.md)

---

## 1. 问题描述

项目导入模块文档定义了 **11 步**导入流水线，各步骤的服务文件大多已实现，但 **没有被串联到导入路由中**。

当前 `POST /api/v1/projects/import` 路由的实际行为：

```
请求进入
  ├── 1. cloneOrCopyProject()     ✅ 同步执行
  ├── 2. scanDirectory()           ✅ 同步执行
  ├── 3. detectTechStack() (内联)  ✅ 同步执行（但用了简化版）
  ├── 4. 硬编码4步ImportTask       ← 仅创建对象，step4永远pending
  └── 立即返回 201               ← 剩余8步从未执行
```

**后果：**
- 前端进度页轮询永远收不到 `done`，用户看到无限等待
- `docParser`、`codeAnalyzer`、`summaryGenerator`、`skillGenerator`、`featureMap`、`docConverter`、`vectorStore`、`gitHistoryAnalysis` 等服务从未在导入流程中被调用
- `importOrchestrator.ts` 已编写完整的状态管理逻辑，但未被 `project.ts` 导入

## 3. 改造目标

### 3.1 重写 importOrchestrator.ts

将编排器从「状态管理器」升级为「流程执行器」：

```typescript
// 核心改动：每个步骤绑定真实的服务调用
const STEP_EXECUTORS: Record<ImportStepName, StepExecutor> = {
  clone:          (ctx) => cloneOrCopyProject(ctx.source, ctx.url, ctx.localPath),
  scan:           (ctx) => scanDirectory(ctx.projectPath),
  detectTech:     (ctx) => detectTechStack(ctx.projectPath),
  parseDocs:      (ctx) => parseDirectory(ctx.projectPath),
  analyzeCode:    (ctx) => analyzeCodeArchitecture(ctx.projectPath),
  detectBuild:    (ctx) => detectBuildMechanism(ctx.projectPath),
  buildSummary:   (ctx) => generateSummary(ctx.projectPath, ctx.techStack),
  generateFeatureMap: (ctx) => generateFeatureMap(ctx.projectPath, ctx.projectId, ctx.projectName),
  generateSkills: (ctx) => generateSkills(ctx.projectName, ctx.projectPath, ctx.architecture),
  convertDocs:    (ctx) => convertDocuments(ctx.projectPath, ctx.projectId),
  vectorize:      (ctx) => vectorizeProject(ctx.projectPath, ctx.projectId, ctx.documents),
  gitHistory:     (ctx) => analyzeGitHistory(ctx.projectPath),
  done:           () => Promise.resolve(),
};
```

### 3.2 改造 POST /import 路由

```
请求进入
  ├── 1. 参数校验
  ├── 2. 通过 orchestrator 创建导入任务
  ├── 3. 立即返回 { project, task }（status: processing）
  └── 4. 后台异步执行：
        orchestrator.runAllSteps(taskId)
          ├── step 1: clone
          ├── step 2: scan
          ├── step 3: detectTech
          ├── ...
          └── step N: done
        每步完成后更新 task.steps[i].status
```

### 3.3 步骤与文档定义对齐

将 `ALL_STEPS` 从当前 8 步扩展为与文档一致的 11 步（+2 辅助步 = 13 步），或合并为合理的步骤数。建议以文档 11 步为准。

## 4. 实施步骤

### 步骤 1：重构 importOrchestrator.ts

- 定义步骤执行上下文 `ImportContext`（包含 projectPath, projectId, techStack 等中间结果）
- 每个步骤接收上下文、产出结果写回上下文
- 添加 `async runAllSteps(taskId: string): Promise<void>` 方法
- 每步执行前调用 `executeStep()`、完成后调用 `completeStep()`
- 异常时标记 `error` 并停止

### 步骤 2：改造 project.ts POST /import

- 移除内联的 `detectTechStack()`、`detectBuildTool()` 函数
- 改为：调用 orchestrator 创建任务 → 立即响应 → `setImmediate` 或 `Promise` 启动后台流程
- 导入并使用 `techDetector.ts` 的 `detectTechStack()`

### 步骤 3：对齐前后端步骤定义

- 后端 `ALL_STEPS` 与前端 `STEP_LABELS` 对齐为同一组步骤名
- 前端 `renderStep3` 不再需要硬编码 fallback 步骤列表

### 步骤 4：添加错误恢复机制

- 单步失败不应导致整个导入不可用
- 对非关键步骤（如 vectorize、generateSkills）失败时标记 `warning` 但继续
- 关键步骤（clone、scan）失败时整体标记 `error`

| # | 验收项 | 验证方式 |
