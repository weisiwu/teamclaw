## L6. Husky Git Hooks 生效

### 现状

`package.json` 中配置了完整的 lint-staged 和 husky：

```json
{
  "scripts": {
    "prepare": "husky"
  },
  "lint-staged": {
    "*.{js,jsx,ts,tsx}": ["prettier --write", "eslint --fix"],
    "*.{json,md,mdx,css}": ["prettier --write"]
  }
}
```

但 `.husky/` 目录为空，没有 `pre-commit` hook 文件。这意味着：
- `npm install` 时 `prepare` 脚本虽然执行了 `husky`，但没有 hook 被注册
- 提交代码时 `lint-staged` 不会被触发
- 格式不规范的代码可以直接提交

### 操作步骤

```bash
# 1. 初始化 husky（如果 .husky 目录不存在）
npx husky init

# 2. 创建 pre-commit hook
echo "npx lint-staged" > .husky/pre-commit

# 3. 确保可执行
chmod +x .husky/pre-commit

# 4. 验证
git add .
git commit -m "test: verify husky hook"
# 应看到 prettier 和 eslint 自动执行
```

### 最终文件内容

**`.husky/pre-commit`**：

```bash
npx lint-staged
```

### 涉及文件

| 操作 | 文件 |
|------|------|
| 新建 | `.husky/pre-commit` |

### 验收

| # | 验收项 | 验证方式 |
|---|--------|---------|
| 1 | `.husky/pre-commit` 文件存在 | `cat .husky/pre-commit` |
| 2 | 提交时自动执行 prettier | 故意加错格式后 commit |
| 3 | ESLint 错误阻止提交 | 故意引入 lint 错误后 commit |

## L7. `server/dist/` 移出版本控制

### 现状

`server/dist/` 是 TypeScript 编译输出目录，包含 **143 个文件**，已提交到 Git 仓库。

构建产物不应进入版本控制：
- 增大仓库体积
- 每次编译都产生大量 diff
- 不同开发者的构建环境不同可能导致冲突
- CI/CD 应在部署时自行编译

### 当前 .gitignore

```
# production
/build
/dist          ← 只忽略了根目录的 dist，没有 server/dist
```

### 操作步骤

```bash
# 1. 在 .gitignore 中添加 server/dist
echo "server/dist/" >> .gitignore

# 2. 从 Git 追踪中移除（不删除文件本身）
git rm -r --cached server/dist

# 3. 提交
git add .gitignore
git commit -m "chore: remove server/dist from version control"
```

### 涉及文件

| 操作 | 文件 |
|------|------|
| 修改 | `.gitignore`（添加 `server/dist/`） |
| Git 操作 | `git rm -r --cached server/dist` |

### 验收

| # | 验收项 | 验证方式 |
|---|--------|---------|
| 1 | `server/dist/` 在 `.gitignore` 中 | `grep "server/dist" .gitignore` |
| 2 | `git status` 不再显示 `server/dist/` 下的文件 | 修改 dist 后 `git status` 无变化 |
| 3 | `server/dist/` 文件仍然存在于磁盘 | `ls server/dist/` |
| 4 | `npm run build`（在 server 中）仍然正常 | 编译输出到 `server/dist/` |

## L8. 清理空目录

### 现状

| 目录 | 状态 | 建议 |
|------|------|------|
| `data/` | 空目录，无任何文件 | 删除 |
| `coverage/` | 空目录 | 删除（由测试工具自动生成） |

注意：以下目录虽然显示 0 items 但是正常的（不应删除）：
- `.git/` — Git 仓库元数据
- `.husky/` — 即将在 L6 中生成 hook
- `.next/` — Next.js 构建缓存（`.gitignore` 已忽略）
- `.vercel/` — Vercel 配置（`.gitignore` 已忽略）
- `node_modules/` — 依赖目录

### 操作步骤

```bash
# 1. 删除空目录
rm -rf data/
rm -rf coverage/

# 2. 确保 .gitignore 已涵盖 coverage（已存在）
grep "coverage" .gitignore
# 输出: /coverage  ← 已存在，OK

# 3. 如果未来需要 data 目录，在代码中动态创建
# mkdir -p data/  在需要时由代码执行
```

### 涉及文件

| 操作 | 目录 |
|------|------|
| 删除 | `data/` |
| 删除 | `coverage/` |

### 验收

| # | 验收项 | 验证方式 |
|---|--------|---------|
| 1 | `data/` 目录不存在 | `ls data/` 报错 |
| 2 | `coverage/` 目录不存在 | `ls coverage/` 报错 |
| 3 | `npm run test:coverage` 后 `coverage/` 自动重建 | 运行测试后检查 |
| 4 | 项目正常启动和运行 | `npm run dev` |
