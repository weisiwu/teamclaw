## L1. 删除冗余 `next.config.mjs`

### 现状

项目根目录同时存在两个 Next.js 配置文件：

| 文件 | 大小 | 内容 |
|------|------|------|
| `next.config.js` | 31 行 | 包含 `output: 'standalone'`、`optimizePackageImports`、`removeConsole`、`modularizeImports` 等完整优化配置 |
| `next.config.mjs` | 117 字节 | 疑似残留，内容极少 |

Next.js 的配置加载优先级：`.mjs` > `.js`。这意味着 `next.config.js` 中的所有优化配置**可能被覆盖**。

### 操作步骤

1. 确认 `next.config.mjs` 内容无有用配置
2. 删除 `next.config.mjs`
3. 验证 `npm run build` 正常

### 涉及文件

| 操作 | 文件 |
|------|------|
| 删除 | `next.config.mjs` |

### 验收

- `npm run build` 成功
- `standalone` 输出目录生成
- `console.log` 在生产构建中被移除（检查构建产物）

## L2. 前端暗色模式类名去重

### 现状

多个组件中同时写了两个冲突的 dark mode 类：

```html
class="text-gray-500 dark:text-gray-400 dark:text-slate-400"
```

`dark:text-slate-400` 覆盖 `dark:text-gray-400`，后者永远无效。

### 受影响文件

```bash
# 查找命令
grep -rn "dark:text-gray-400 dark:text-slate-400" app/ components/ --include="*.tsx"
```

已知受影响文件：
- `app/page.tsx` — 至少 3 处
- 其他组件待全局搜索确认

### 操作步骤

1. 全局搜索 `dark:text-gray-400 dark:text-slate-400`
2. 统一为 `dark:text-slate-400`（与项目 dark theme 色板一致）
3. 同时检查是否有其他类似冗余（如 `dark:bg-gray-800 dark:bg-slate-800`）

### 统一规范

| 场景 | 推荐类名 | 不推荐 |
|------|---------|-------|
| 暗色次要文字 | `dark:text-slate-400` | `dark:text-gray-400 dark:text-slate-400` |
| 暗色背景 | `dark:bg-slate-900` | `dark:bg-gray-900` |
| 暗色卡片背景 | `dark:bg-slate-800` | `dark:bg-gray-800` |
| 暗色边框 | `dark:border-slate-700` | `dark:border-gray-700` |

> 项目整体 dark 主题基于 `slate` 色系（见 `AppLayout.tsx` 的 `dark:bg-slate-900`），应统一用 `slate`。

### 涉及文件

| 操作 | 文件 |
|------|------|
| 修改 | `app/page.tsx` + 全局搜索出的其他文件 |

### 验收

- `grep -rn "dark:text-gray-400 dark:text-slate-400"` 返回 0 结果
- 暗色模式下页面文字颜色统一

## L4. `lib/api/types.ts` 拆分为类型和常量

### 现状

`lib/api/types.ts`（1227 行）文件名暗示只有类型定义，但实际包含：

| 内容 | 示例 | 行数 |
|------|------|------|
| 纯 TypeScript 类型 | `interface Task { ... }`、`type TaskStatus = ...` | ~600 |
| 运行时常量数组 | `TASK_STATUS_OPTIONS`、`MEMBER_ROLE_OPTIONS` | ~200 |
| 运行时映射对象 | `STATUS_BADGE_VARIANT`、`MODEL_PRICING` | ~200 |
| 数据配置 | 模型单价 `{ inputPrice: 0.03, outputPrice: 0.06 }` | ~200 |

### 问题

- 纯类型（编译时擦除）和运行时常量混在一起
- 引用一个类型时会导入整个文件的常量到 bundle
- 文件名 `types.ts` 具有误导性

### 操作步骤

拆分为两个文件：

| 新文件 | 内容 | 说明 |
|-------|------|------|
| `lib/api/types.ts` | 所有 `interface`、`type`、`enum` | 编译时擦除，不影响 bundle |
| `lib/api/constants.ts` | 所有 `const` 对象和数组 | 运行时常量，按需导入 |

`constants.ts` 内容示例：

```typescript
import type { TaskStatus, MemberRole, CronStatus, ModelType, ModelPricing } from './types';

export const TASK_STATUS_OPTIONS = [ ... ] as const;
export const STATUS_BADGE_VARIANT: Record<TaskStatus, string> = { ... };
export const MEMBER_ROLE_OPTIONS = [ ... ] as const;
export const MODEL_PRICING: Record<ModelType, ModelPricing> = { ... };
// ...
```

然后全局更新 import 路径：

```bash
# 查找引用常量的文件
grep -rn "TASK_STATUS_OPTIONS\|STATUS_BADGE_VARIANT\|MEMBER_ROLE_OPTIONS\|MODEL_PRICING" app/ components/ --include="*.tsx" --include="*.ts"
```

### 涉及文件

| 操作 | 文件 |
|------|------|
| 修改 | `lib/api/types.ts`（移除常量，仅保留类型） |
| 新建 | `lib/api/constants.ts`（接收移出的常量） |
| 修改 | 所有引用常量的组件和页面文件 |

### 验收

- `lib/api/types.ts` 中无 `export const` 语句
- `lib/api/constants.ts` 中无 `export interface` / `export type` 语句
- `npm run build` 无 TypeScript 错误
- 所有页面功能正常
