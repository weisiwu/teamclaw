# 【P0】大量页面功能不可用与 API 报错排查修复

## 背景
本地运行项目后，大量页面点击进去就报错或功能不可用。API 返回非 JSON 内容（如 HTML 404）导致前端 JSON.parse 崩溃。大量接口可能因为数据库表未初始化、路由未注册、或服务端启动顺序问题而不可用。

## 目标
- 逐页巡检所有前端页面，记录报错页面和报错原因
- 修复所有因路由缺失、表未创建、服务未初始化导致的 API 错误
- 确保所有侧边栏入口对应的页面至少可以正常加载（即使无数据也不报错）
- 前端请求非 JSON 响应时不崩溃

## 技术方案
1. 编写页面健康检查脚本，自动访问所有路由并记录 HTTP 状态码
2. 修复返回 404/500 的 API 端点
3. 前端 fetch 封装中增加 Content-Type 检查，非 JSON 时不尝试 parse
4. 数据库迁移补全缺失的表
5. 服务端启动时检查关键表是否存在

## 实现文件
- `server/src/routes/*.ts` — 修复返回错误的路由
- `server/src/db/migrations/*.ts` — 补全缺失表迁移
- `app/lib/api.ts` 或各页面 fetch — 增加响应类型检查
- `scripts/healthcheck.sh` — 新建页面健康检查脚本

## 验证方式
1. 所有侧边栏页面可正常打开（无白屏/崩溃）
2. API 返回 HTML 时前端展示友好提示而非 JSON parse 错误
3. 服务端启动日志无表缺失警告
4. healthcheck 脚本所有路由返回 200
5. **至少 5 轮修改验证**：路由修复→迁移补全→前端容错→健康检查→全页面巡检
