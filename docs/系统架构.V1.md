## 实现顺序与阶段说明

整个系统按 **6 个阶段**、**严格按依赖关系从底层到上层** 的顺序实现。后一阶段依赖前一阶段的产出，禁止跳阶段开发。

```
阶段 1：基础设施        阶段 2：数据输入        阶段 3：角色与编排
┌──────────────┐      ┌──────────────┐      ┌──────────────┐
│ ① 技术选型    │─────→│ ③ 项目导入    │─────→│ ④ 人员与权限  │
│ ② 部署方案    │      │              │      │ ⑤ Agent 编排  │
└──────────────┘      └──────────────┘      └──────┬───────┘
                                                    │
                  ┌─────────────────────────────────┘
                  ▼
阶段 4：核心流程        阶段 5：产出与辅助       阶段 6：管理界面
┌──────────────┐      ┌──────────────┐      ┌──────────────┐
│ ⑥ 消息机制    │─────→│ ⑧ 版本管理    │─────→│ ⑩ 后台管理平台│
│ ⑦ 任务机制    │      │ ⑨ 辅助能力    │      │  （全量集成）  │
└──────────────┘      └──────────────┘      └──────────────┘
```

### 阶段 1：基础设施（① 技术选型 → ② 部署方案）

**目标**：确定技术栈，搭好开发和部署环境。

- 先敲定框架、语言、数据库、模型分级等技术决策
- 再完成部署架构设计、本地开发环境搭建、目录结构初始化
- **产出**：可运行的空项目骨架 + CI/CD 基础配置

**前置依赖**：无（起点）

### 阶段 2：数据输入（③ 项目导入模块）

**目标**：实现项目代码的导入、解析和记忆生成。

- 实现 URL 下载 / 本地上传两种数据源接入
- 完成 11 步导入流程：文件扫描、技术栈识别、文档解析、代码架构梳理、摘要生成、RAG 向量化、Skill 自动生成
- **产出**：项目记忆库 + Skills + 向量数据库

**前置依赖**：阶段 1（需要技术栈和部署环境就绪）

### 阶段 3：角色与编排（④ 人员与权限 → ⑤ Agent 编排）

**目标**：建立角色权限体系，配置 Agent 团队。

- 先实现角色体系（管理员/副管理员/普通员工）和权限规则
- 再配置 Agent 团队成员（main/pm/coder/reviewer）、等级指派规则、PM 交互协议
- **产出**：可用的角色权限系统 + Agent 团队配置

**前置依赖**：阶段 2（Agent 需要读取项目记忆和 Skills）

### 阶段 4：核心流程（⑥ 消息机制 → ⑦ 任务机制）

**目标**：打通从消息到任务的完整流程。

- 先实现消息接入（微信/飞书）、消息队列、优先级抢占
- 再实现任务生命周期（创建 → 进行中 → 测试 → 完成）、任务流转、任务记忆化
- **产出**：完整的消息驱动任务执行链路

**前置依赖**：阶段 3（消息需要身份鉴权，任务需要 Agent 执行）

### 阶段 5：产出与辅助（⑧ 版本管理 → ⑨ 辅助能力）

**目标**：管理代码产出，提供辅助查询能力。

- 实现 Git Tag 自动创建、版本面板、构建打包、产物下载、版本回退
- 实现辅助能力：项目查询、文档浏览、文件下载等自助功能
- **产出**：版本管理系统 + 辅助能力工具集

**前置依赖**：阶段 4（版本管理需要任务完成后触发；辅助能力需要项目记忆和任务数据）

### 阶段 6：管理界面（⑩ 后台管理平台）

**目标**：将所有模块集成到统一的 Web 管理界面。

- 后台管理平台是所有模块的**可视化入口**，需要在各模块基本就绪后做最终集成
- 包括：项目导入向导、任务管理面板、版本管理面板、辅助能力管理、定时任务管理、Token 消费统计
- **产出**：完整可用的 Web 后台

**前置依赖**：阶段 1-5 全部完成

> **注意**：后台管理平台的 UI 骨架可以在阶段 1 随项目骨架一起搭建，但各功能页面应随对应模块的完成逐步集成，最终在阶段 6 做全量联调和样式验收。

## Agent 实现指引规则

以下规则适用于所有模块的实现过程，Agent 在编码时必须严格遵守。

### 规则 1：按模块文档中的「实现任务清单」逐项执行

每个模块文档都包含一个**实现任务清单**，列出了该模块所有需要创建的文件、API、组件和配置。Agent 必须：

1. 按清单顺序逐项实现，每完成一项在代码中留下对应注释
2. 不跳过任何一项，不自行增删任务
3. 遇到不确定的实现细节时，以文档描述为准

### 规则 2：每个功能必须同时完成前后端

一个功能 = 后端 API + 前端页面/组件 + 联调验证。禁止只写后端不写前端，或只写前端用 mock 数据。

### 规则 3：禁止 mock 数据

所有数据必须来自真实项目仓库（默认 https://github.com/weisiwu/teamclaw ）。数据源在 [项目导入模块](./modules/项目导入模块.md) 中配置。

### 规则 4：每个功能完成后必须检测 UI 样式

实现任何有界面的功能后，必须在浏览器中验证：
- 布局是否正确（间距、对齐、响应式）
- 配色是否符合设计系统
- 交互状态是否完整（hover、loading、empty、error）
- 不允许"逻辑跑通就算完成"

### 规则 5：严格遵守阶段依赖

- 当前阶段未完成前，不得开始下一阶段
- 每个阶段的验收标准在对应模块文档的「验收标准」章节中定义
- 阶段验收通过后才能推进

### 规则 6：文件组织规范

```
apps/teamclaw/
├── server/                    # 后端服务
│   ├── src/
│   │   ├── routes/            # API 路由，按模块分文件
│   │   ├── services/          # 业务逻辑
│   │   ├── models/            # 数据模型
│   │   ├── middleware/        # 中间件（鉴权、日志等）
│   │   └── utils/             # 工具函数
│   ├── migrations/            # 数据库迁移
│   ├── package.json
│   └── tsconfig.json
├── dashboard/                 # 前端后台
│   ├── src/
│   │   ├── pages/             # 页面组件，按模块分目录
│   │   ├── components/        # 通用组件
│   │   ├── hooks/             # 自定义 Hooks
│   │   ├── services/          # API 调用封装
│   │   ├── stores/            # 状态管理
│   │   └── types/             # TypeScript 类型
│   ├── package.json
│   └── vite.config.ts
└── docs/                      # 当前文档目录
```

### 规则 7：API 规范

- 所有 API 使用 RESTful 风格，路径前缀 `/api/v1/`
- 响应格式统一：`{ "code": 0, "data": {...}, "message": "ok" }`
- 错误响应：`{ "code": <错误码>, "data": null, "message": "错误描述" }`
- 分页参数：`?page=1&pageSize=20`，返回 `{ "list": [...], "total": 100 }`

### 规则 8：每个模块的实现产出

每个模块实现完成后，必须产出以下内容：

| 产出 | 说明 |
|------|------|
| **后端 API** | 路由文件 + Service 文件 + 类型定义 |
| **前端页面** | 页面组件 + API 调用封装 + 类型定义 |
| **数据库迁移** | 如有新表或字段变更 |
| **单元测试** | 至少覆盖核心业务逻辑 |
| **验收截图/说明** | 证明功能可用且样式正确 |

## 1. 整体框架

### 1.1 核心定位

TeamClaw 是一个**业务接入框架**，核心能力：
- 业务代码导入 → 自动解析 → 生成记忆
- 群聊消息驱动 → 任务调度 → Agent 协作执行
- 版本管理 → 构建交付 → 后台可视化

### 1.2 系统组成

```
┌─────────────────────────────────────────────────────────┐
│                    Web 后台管理平台                       │
│  职责：项目导入、任务管理、版本管理、配置、监控              │
└────────────────────┬────────────────────────────────────┘
                     │ 调用
                     ▼
┌─────────────────────────────────────────────────────────┐
│                   三层架构核心                            │
├─────────────────────────────────────────────────────────┤
│  Layer 1: WorkBuddy（入口层）                            │
│  核心职责：                                               │
│  1. 数据转换：微信/飞书消息 → 统一任务格式                  │
│  2. 角色暴露：将 Layer3 的 main、pm 暴露到群聊             │
│  3. 身份鉴权、消息队列、优先级排序                          │
│  模型：轻量模型（DeepSeek-Chat / Kimi）                   │
├─────────────────────────────────────────────────────────┤
│  Layer 2: MiddlewareClaw（调度层）                       │
│  核心职责：                                               │
│  1. 任务队列管理：统一管理所有任务的生命周期                 │
│  2. 记忆优化：每条任务后续都进行记忆化处理                   │
│  3. 上下文补充：从项目记忆中召回相关上下文                   │
│  4. 任务拆解、业务调度、Prompt 生成                        │
│  模型：中等模型（GPT-4o-mini / DeepSeek-V3）              │
├─────────────────────────────────────────────────────────┤
│  Layer 3: Agents OpenClaw（执行层）                      │
│  核心职责：                                               │
│  1. 静默执行：接收 Layer2 的精炼 Prompt，安静地拆分任务     │
│  2. 具体实现：代码编写、文件操作、Git 操作、测试             │
│  3. 结果上报：将执行结果返回 Layer2                        │
│  模型：强模型（Claude Sonnet / GPT-4o）                   │
│  角色：main、pm（暴露在群）、coder1、coder2、reviewer（隐藏）│
└─────────────────────────────────────────────────────────┘
```

### 1.3 数据流

```
用户群聊消息（微信/飞书）
    │
    ▼
Layer1 (WorkBuddy)
    ├── 数据转换：消息 → 任务格式
    ├── 角色暴露：用户感知到 main、pm 的存在
    ├── 身份鉴权：判断用户角色和权限
    └── 任务生成 → tasks/inbox/{task_id}_{priority}.json
    │
    ▼
Layer2 (MiddlewareClaw)
    ├── 任务队列管理：入队、优先级排序、抢占
    ├── 记忆优化：召回项目记忆、压缩上下文
    ├── 上下文补充：RAG 检索相关代码/文档
    ├── 任务拆解：大任务 → 子任务
    └── Prompt 生成 → tasks/dispatched/{task_id}.json
    │
    ▼
Layer3 (Agents OpenClaw)
    ├── main：任务分配、质量把控
    ├── pm：需求细化、群消息收集
    ├── coder1/coder2：代码编写
    ├── reviewer：代码审查
    └── 执行结果 → tasks/completed/{task_id}_result.json
    │
    ▼
Layer2 汇总结果 → Layer1 回传群聊
```

### 1.5 关键设计原则

1. **角色暴露机制**：只有 main 和 pm 在群聊中可见，避免用户直接给低级 Agent 下达混乱指令
2. **任务队列管理**：Layer2 统一管理任务队列，支持优先级、抢占、挂起
3. **记忆优化**：每个任务完成后都要记忆化，方便后续检索和复用
4. **上下文补充**：通过 RAG 从项目记忆中召回相关信息，减少重复询问
