我希望利用基于openclaw开发一个接入业务的框架，业务代码通过框架导入后。框架通过多种机制，去解析业务代码，预先生成记忆，调度任务、消息队列、agent调度等能力，去完成项目方的目标
项目导入
1. 模型能力是否要分级，分成几级？
 用最厉害的模型，去干最核心的任务    
2. 已有的项目是什么样子的？
 有各种文档，比如 ppt/word/excel/text 还有各种代码文件
 如何让AI能快速解析这些文件，并且能够理解文件的内容？
3. 对已有项目进行理解后，是要生成RAG还是什么，才能加速后续的理解？
4. 阅读完毕项目后，是否可以自动生成一些skill？放在特定的目录，供所有的agent使用？
5. 默认是否需要提供一些扩展、tool，增强能力，方便导入？
6. 梳理代码架构
7. 了解代码打包机制、编译机制
8. 项目需要有长期代码记忆
9. 项目上传后，立刻开始搜索项目中特定类型的文档，将这些文档转换为在线文档，供浏览
10. 生成定位文件，对整个页面的功能做拆分
11. 基于历史改动，轮次对话，有 记忆共享，反馈增强
核心: 你需要思考，如何讲一个很大的上下文压缩成一个摘要，然后，如何通过摘要能快速对应原文，然后再做进一步的搜索
这里还涉及，项目内容补充。比如，我有个启动网站，内网部署，启动后，让指定一个路径，我去读取路径下的内容。开始做初始化，后面内容有更新，通过定时任务去刷新，或者通过手动强制刷新去执行

人员关系导入
1. 快速确定群聊人员等级/角色
 是不是可以通过后台配置的方式，来确定谁可以访问？
2. 禁止私聊
3. 群聊回话记录做保存，并且向量化
4. 提供快速搜索群聊的 search 工具
5. 不同等级/角色的人，可以指派的机器人不同。
6. 人员也需要形成记忆，保存到数据库
核心: 同样是增加一层记忆，然后设计一套agent遵循的规范，这套规范是不是可以做成代码？直接固定化的执行

agent导入
1. 团队成员: main、coder1、coder2、reviewer、pm
2. 分工： 
 main(主管): 任务分配，不做具体工作
 coder1(程序员1号): 代码编写
 coder2(程序员2号): 代码编写
 reviewer(代码审查): 代码审查，问题修复
 pm(产品经理): 帮主管对需求做拆分和细化，收集整理群里的消息
 他应该是先提出不多于N个的问题，等待回复，回复他了，他就记住。全部回复完毕，就开始进行下一步
3. agent虽多，但不需要所有agent都在群里，默认只有 pm 和 main 在群
 避免其他人给agent指定乱起八遭的任务

消息机制
1. 每个明确AT机器人的消息，都算作任务的开始
2. 语音消息
 暂不支持
3. 文件信息，全部存储起来，并做文件理解，方便后续说话
 只看部分类型的文档
4. 表情信息，直接无视
5. 在agent可以做出响应前的信息，全部存储起来，形成消息队列，立刻给予回复，消息已收到，稍后处理，等待agent响应后，再做处理
6. 消息队列需要支持动态搞优先级，比如老板说的话，就要清空前面所有的任务，只做老板的话，并且发送回复消息，告知其他人的任务已被放弃。
任务机制
1. 任务由消息中来
2. 开启任务的方法是，当有人@main agent，main agent 需要接下来展开对话，同用户进行讨论。
3. 明确任务的方法是，当讨论到一定程度，老板觉得没问题，或者主管主动询问老板，得到肯定答复，形成任务
4. 形成任务后，需要在群里立刻由pm同步任务信息，并形成一条连接，可以随时查看任务状态
5. 任务形成后，有这些生命周期: 进行中、测试中、已完成、已取消
6. 任务一经委派，就是进行中
7. 任务开发完毕，由main agent在群里发布消息，告知所有人，任务已经完成，消息发送完毕后，任务进入测试中
8. 任务有任务详情页，打开里面有任务详细信息，和完成、取消按钮，点击对应按钮，进入对应状态
9. 任务需要有长期记忆，并且保存在向量数据库中。

协作机制
1. agent分等级: lv1 lv2 lv3， lv3 最高级，lv1最低，高级可指派低级
2. 员工分角色: 管理员、副管理员，普通员工。不同角色权重不同，管理员权重最高，普通员工权重最低。
3. 员工中只有 管理员和副管理员可以同主管交流，发布任务
4. 所有角色的员工都可以找 pm 沟通
5. 只有管理员和副管理员可以和main沟通，其他人的沟通 main 回复: 正在忙，抱歉
6. 管理员和副管理员可以和pm沟通任何事情，发出任何指令，普通员工只能问系统提供的辅助能力
7. 管理员发出的指令优先级高于副管理员，副管理员发出的指令优先级高于普通员工

版本机制
1. 对于所有的特定版本，都生成一个git tag
2. 提供一个面板，面板上有所有git tag 的信息和提交内容信息
3. 可以直接打包。提供重新打包和立刻下载按钮
4. 提供打包产物的下载方式，
5. 提供恢复机制
6. 提供分支机制
7. 任务完成后，自动版本升级
8. 提供版本追踪制度，比如消息截图
9. 版本也需要生成摘要，存放到数据库中

辅助能力
1. 查看项目文档库
 群里所有的文档消息都会被上传，生成文档
2. 查看项目详情
3. 查看任务列表
4. 查看具体任务
5. 查看产物列表
6. 下载文件
7. 有权限的人员，直接操作后台的各种配置

后台配置
1. 网页版本的配置平台
2. 需要先初始化，分别有几项：导入项目
3. 任务管理: 任务列表、任务详情，任务完成，任务取消
4. 版本管理: 版本列表，版本详情页，创建分支，指定主分支，构建详情，重新构建，下载产物
5. 辅助能力管理: 辅助能力列表，能力开关
6. 定时任务管理，查看所有定时任务，添加定时任务，定时任务只支持prompt，口头描述后，创建prompt
7. 项目文档管理，可查看所有文档
8. token消费查看，比如一天工作消费了多少token

流程
用户 -> workbuddy -> 整理消息 -> enhanced gateway -> channel/tui/browser -> openclaw gateway -> skill/tools/self-skills/self-tools
enhanced gateway -> create task
enhanced gateway -> fetch data from db
workbuddy -> middlewareclaw -> agents openclaw
workbuddy: 接入微信消息
middlewareclaw: 处理项目信息，业务调度，压缩消息，产出指令
agents openclaw: 处理实际工作人的调度
